Step 0 — LoRA 的数学基础：为什么低秩有效

Aghajanyan et al. (2021) 发现：预训练模型有很低的内在维度（intrinsic dimensionality）。这意味着 fine-tuning 时的权重更新 ΔW 集中在一个低维子空间里。

核心数学直觉：
对权重更新矩阵 ΔW 做 SVD 分解：ΔW = U Σ V^T。
实验表明，Σ 中前 k 个奇异值占据了绝大部分能量（>95%），其中 k 远小于 min(d_in, d_out)。

因此 ΔW 可以近似为：ΔW ≈ U_k Σ_k V_k^T，这是一个秩为 k 的矩阵。

LoRA 直接参数化这个低秩结构：ΔW = B · A，其中 B ∈ R^{d_out × r}，A ∈ R^{r × d_in}，r 就是 rank。
为什么不直接训练 ΔW？
• ΔW ∈ R^{d_out × d_in}，参数量 = d_out × d_in（例如 4096 × 4096 = 16M）
• B·A 参数量 = (d_out + d_in) × r（例如 (4096 + 4096) × 16 = 131K）
• 参数量减少 122 倍，显存和计算同步减少
• 代价：限制了 ΔW 的秩最多为 r，但实验表明这个限制对性能影响很小

Step 1 — 显存精确计算

以 LLaMA-2 7B 为例，精确计算 GPU 显存占用：

Full Finetune (~56 GB)
• 参数 (fp16): 7B × 2 bytes = 14 GB
• 梯度 (fp16): 14 GB
• AdamW 状态 (fp32 m + v): 7B × 4 × 2 = 56 GB... 不，是 2 × 14 = 28 GB
• 合计: 14 + 14 + 28 = ~56 GB
LoRA (~14.1 GB)
• 冻结参数 (fp16): 14 GB（只存权重，不存梯度/优化器）
• LoRA 参数 (fp16): ~16 MB
• LoRA 梯度 (fp16): ~16 MB
• LoRA AdamW (fp32): ~32 MB
• 合计: ~14.1 GB
QLoRA (~3.6 GB)
• 冻结参数 (NF4 4-bit): 7B × 0.5 bytes = 3.5 GB
• LoRA 参数 (fp16): ~16 MB
• LoRA 梯度 + 优化器: ~48 MB
• 合计: ~3.6 GB
注意：以上不包含 activations（前向传播中间结果）。Activations 的显存取决于 batch size × sequence length × hidden_size × num_layers，通常是显存的最大部分。gradient_checkpointing 可以用时间换空间来减少 activations 显存。

Step 2 — 混合精度中的数值分析

LoRA 的前向计算涉及多种精度的混合运算：

# LoRA forward 的核心逻辑（简化） def forward(self, x): # base_layer 在 fp16/bf16 result = self.base_layer(x) # W·x (fp16) # LoRA 路径：x 先转到 LoRA 的 dtype lora_x = x.to(self.lora_A.weight.dtype) # 确保 dtype 匹配 lora_out = self.lora_B(self.lora_A(lora_x)) # B(A(x)) (fp16) result += self.scaling * lora_out # scaling = alpha/r return result
scaling = alpha / r 的数值影响：
• alpha=32, r=16 → scaling = 2.0：LoRA 的输出被放大 2 倍
• alpha=16, r=16 → scaling = 1.0：不缩放
• alpha=8, r=16 → scaling = 0.5：LoRA 的输出被缩小

如果 scaling 太大 → LoRA 输出主导结果，训练不稳定
如果 scaling 太小 → LoRA 的更新效果太弱，学不到东西
QLoRA 的精度链路：
1. 基础权重存储：NF4 (4-bit) → 前向时反量化到 fp16/bf16
2. LoRA A, B 参数：fp16/bf16
3. LoRA 梯度：fp16/bf16
4. LoRA 优化器状态：fp32（AdamW 的 m 和 v）
5. 注意：反量化是逐块（blockwise）的，引入量化误差，但实验表明影响很小

Step 3 — LoRA 变体：DoRA

DoRA (Weight-Decomposed Low-Rank Adaptation) 将权重分解为大小（magnitude）和方向（direction）两个分量，LoRA 只修改方向：

DoRA 的数学：
1. 将预训练权重 W 分解为：W = m · V / ||V||_c（按列归一化）
  其中 m 是每列的范数（magnitude），V/||V|| 是方向（direction）
2. 用 LoRA 修改方向：V' = V + B · A
3. magnitude m 作为独立可训练参数
4. 最终输出：W' · x = m · (V + B·A) / ||V + B·A||_c · x
from peft import LoraConfig # 启用 DoRA — 只需加一个参数 config = LoraConfig( r=16, lora_alpha=32, target_modules=["q_proj", "v_proj"], use_dora=True, # 启用 DoRA ) # 其余用法和普通 LoRA 完全一样
DoRA 的优势：
• 解耦了 magnitude 和 direction 的学习，训练更稳定
• 在相同 rank 下通常比 LoRA 获得更好的性能
• 额外参数量极小（只多了一个 magnitude 向量 m）
• 计算开销略高于 LoRA（需要列归一化），但不显著

Step 4 — LoRA 变体：rsLoRA, LoRA+, AdaLoRA

rsLoRA (Rank-Stabilized)
缩放因子改为 alpha / sqrt(r)
原始 LoRA 用 alpha / r，当 r 增大时缩放变小 → 效果下降。
rsLoRA 用 sqrt(r) 使大 rank 也能有效学习。
LoRA+
A 和 B 使用不同学习率：B 用更高 lr。
直觉：B 负责"放大"到输出空间，需要更大步长；
A 负责"压缩"输入，更新幅度小即可。
AdaLoRA (Adaptive LoRA)
动态调整每层的 rank，基于重要性分数：
• 用 SVD 参数化 LoRA：ΔW = P Λ Q（而非 B·A）
• 训练中根据 Λ 中奇异值的大小，修剪不重要的分量
• 效果：重要层分配更高的 rank，不重要层用更低的 rank
• 总参数量保持不变，但分配更合理
# rsLoRA — 只需一个参数 config = LoraConfig(r=64, lora_alpha=64, use_rslora=True) # scaling = alpha / sqrt(r) = 64 / 8 = 8.0（而非 64/64=1.0） # LoRA+ — 通过 optimizer 设置不同 lr # peft 内置支持：loraplus_lr_ratio 参数 config = LoraConfig(r=16, lora_alpha=32) # 在 trainer 中: loraplus_lr_ratio=16 → B 的 lr = 16 × A 的 lr # AdaLoRA from peft import AdaLoraConfig config = AdaLoraConfig( init_r=12, # 初始 rank target_r=8, # 目标 rank（修剪后） tinit=200, # warmup steps（开始修剪前） tfinal=1000, # 修剪结束 step deltaT=10, # 每隔多少 step 修剪一次 )

Step 5 — LoRA 的计算开销分析

LoRA 在 forward 和 backward 中引入的额外计算量：

Forward FLOPs 分析（per token, per layer）：
• 基础层：W·x → 2 × d_in × d_out FLOPs = 2 × 4096 × 4096 = 33.5M FLOPs
• LoRA 路径：A·x → 2 × d_in × r = 2 × 4096 × 16 = 131K FLOPs
                  B·(Ax) → 2 × r × d_out = 2 × 16 × 4096 = 131K FLOPs
• LoRA 额外 FLOPs = 262K / 33.5M = 0.8% overhead

即使对所有 attention 层（q, k, v, o）都加 LoRA，额外开销也不到 3.2%。
Backward 的情况：
• 梯度只需要计算 A 和 B 的梯度（不需要 W 的梯度）
• 但仍然需要通过 base_layer 反向传播（因为 x 需要梯度给前面的层）
• 所以 backward 时间节省主要来自不需要计算和存储 W 的梯度
推理延迟影响：
• 如果不 merge：额外延迟 ~1-3%（因为要多算 A·x 和 B·(Ax)）
• 如果merge 后：零额外延迟（LoRA 已合并到 W 中）
• 所以部署时通常 merge_and_unload 后再部署

Step 6 — merge 的数学分析

merge_and_unload 的核心操作：W' = W + (alpha/r) × B × A

精度分析：
• fp32 下：merge 是精确的，W' = W + ΔW 无损
• fp16 下：存在舍入误差。fp16 只有 ~3.3 位十进制精度（尾数 10 位）
• 误差来源：当 |W| 很大，|ΔW| 很小时，加法发生灾难性消去（catastrophic cancellation）
• 例如：W = 100.0 (fp16)，ΔW = 0.001 → W + ΔW 在 fp16 下仍然 = 100.0
实际影响：
• 对于大多数模型，LoRA 的更新量 ΔW 和 W 的量级差别不超过 2-3 个数量级
• fp16 能正确表示的加法精度足以覆盖大部分情况
• 实验中，merge 前后的输出差异通常在 1e-6 量级 → 可忽略
• 如果非常在意精度：先转 fp32 merge，再转回 fp16
# 如果担心精度，用 fp32 merge model = model.to(torch.float32) merged = model.merge_and_unload() merged = merged.to(torch.float16) # 转回 fp16 做推理 # 或者用 peft 内置的 safe_merge model.merge_adapter(safe_merge=True) # 会检测数值问题

Step 7 — 与分布式训练的集成

LoRA 与分布式训练框架的集成要点：

DeepSpeed ZeRO：
• ZeRO-1：optimizer 状态分片 → LoRA 的 optimizer 状态也被分片，自然兼容
• ZeRO-2：optimizer + gradient 分片 → LoRA 梯度也被分片，自然兼容
• ZeRO-3：param + optimizer + gradient 全分片 → LoRA 参数也被分片到各 GPU
  注意：ZeRO-3 下需要 zero3_init_flag=False 避免 LoRA 参数被过度分片
FSDP (Fully Sharded Data Parallel)：
• LoRA 参数很小（~16MB），分片收益不大
• 关键是 fsdp_auto_wrap_policy 的设置：应按 transformer 层包装
• peft 提供了 fsdp_auto_wrap_policy 相关工具函数
# DeepSpeed ZeRO-2 + LoRA（最常见组合） # ds_config.json: { "bf16": {"enabled": true}, "zero_optimization": { "stage": 2, "offload_optimizer": {"device": "cpu"} // 可选：把优化器放 CPU }, "gradient_checkpointing": true // 推荐：减少 activation 显存 }
常见的高效组合：
• gradient_checkpointing + LoRA + ZeRO-2：最常用，兼容性最好
• QLoRA + ZeRO-3：多 GPU 训练大模型的极致省显存方案
• LoRA + FSDP：PyTorch 原生方案，不需要 DeepSpeed 依赖
• 注意：gradient_checkpointing 和 LoRA 一起使用时需要 model.enable_input_require_grads()

Step 8 — 实际性能 Benchmark

基于 LLaMA-2 7B 在典型指令微调任务上的对比（数值为典型值，具体因硬件/数据而异）：

关键结论：
• 性能（准确率/质量）：LoRA 通常达到 Full FT 的 95-99%，差距很小
• 训练速度：LoRA 和 Full FT 的 tokens/sec 相近（LoRA 开销小），QLoRA 慢 ~10-20%（反量化开销）
• 显存：LoRA 是 Full FT 的 ~1/4，QLoRA 是 Full FT 的 ~1/15
• 收敛时间：LoRA 通常需要稍多的 epochs 才能达到 Full FT 的水平
什么时候 LoRA 可能不够？
• 任务和预训练数据差异很大（如新语言、全新领域）→ 可能需要更高的 rank 或 full finetune
• 需要修改模型行为的大面积改动 → LoRA 的低秩约束可能是瓶颈
• 实践建议：先用 LoRA (r=16-64) 试，如果效果不够再考虑 full finetune
QLoRA 的额外开销来源：
• 每次 forward 都要对 NF4 权重做 blockwise dequantization → fp16
• 这是纯内存带宽操作，在 GPU 上不算慢，但不是零开销
• 典型场景：QLoRA 训练速度是 LoRA 的 80-90%

Step 9 — 总结与设计回顾

五轮核心 insight：
1. 低秩假设在实践中成立：fine-tuning 的权重更新 ΔW 确实近似低秩
2. peft 的架构 — BaseTuner → LoraModel → lora.Linear — 是干净的分层设计
3. adapter dict 模式使得多任务复用成为可能，无需模型拷贝
4. QLoRA 使 7B 模型在消费级 GPU (24GB) 上可训练
5. merge_and_unload 连接了训练（需要 LoRA 结构）和部署（只需要普通模型）
连接到 mmit：
mmit 的 PEFTStrategy / LoRAStrategy 封装了 peft，额外处理：
• 从多模态 Method 中定位 LLM 模块
• 同时管理 projector 的训练和保存
• 提供统一的 wrap_model / save_adapter / load_adapter 接口
• 底层仍然是 peft 的 LoraConfig + get_peft_model + PeftModel
五轮 API 速查：

LoraConfig(r, lora_alpha, target_modules, use_dora, use_rslora, ...) — 配置
get_peft_model(model, config) — 注入 LoRA
model.print_trainable_parameters() — 检查参数量
model.save_pretrained(path) — 保存 adapter
PeftModel.from_pretrained(base, path) — 加载 adapter
model.merge_and_unload() — 合并 LoRA 到 W
model.add_adapter(name, config) — 添加新 adapter
model.set_adapter(name) — 切换 adapter
model.disable_adapter() — 临时关闭 adapter
prepare_model_for_kbit_training(model) — QLoRA 准备
