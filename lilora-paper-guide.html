LoRA in LoRA (LiLoRA)

Parameter-Efficient Architecture Expansion for Continual Visual Instruction Tuning
Che et al. · arXiv 2508.06202 · 2025

1. 这篇论文想解决什么问题？

核心问题：VLM 持续学习时的"灾难性遗忘"
大型视觉语言模型（VLM）需要不断学习新任务（比如先学 VQA，再学图像描述，再学分类...），但学了新任务后，旧任务的能力会严重下降。传统做法是每个任务训一套独立的 LoRA，参数量大且完全隔离，无法共享知识。

现有方法的缺陷

  • SeqLoRA（顺序微调）：直接在同一 LoRA 上接着训 → 旧任务遗忘严重（BWT = -48）
  • DirLoRA（独立 LoRA）：每个任务单独一套 LoRA → 参数多（每任务 357 MB），无知识共享
  • SMoLoRA（混合专家）：较好，但结构复杂，参数量仍高

LiLoRA 的目标

  • 每任务参数量减少 70%
  • 遗忘程度减少 70%
  • 保持甚至超越独立 LoRA 的性能
  • 支持任意数量的新任务扩展

2. 关键发现：CKA 分析

🔍
作者用 CKA（Centered Kernel Alignment）分析不同任务训出来的 LoRA 矩阵，发现：
• A 矩阵（下投影）在不同任务之间 高度相似（CKA > 0.9）
• B 矩阵（上投影）在不同任务之间 差异很大（CKA < 0.3）
→ 结论：A 可以共享，B 需要任务特定。

3. LiLoRA 架构详解

Standard LoRA
x → [d_in] | A → [r × d_in] 下投影 | B → [d_out × r] 上投影 | ΔW = B · A 每个任务独立的 A 和 B 参数量 = d_in×r + d_out×r
LiLoRA (本文)
x → [d_in] | A → [r × d_in] 共享 | ┌────────┐ | | B₀ 共享 B̃i·Ãi 任务i | | └────────┘ | ΔW = (α·B₀ + (1-α)·B̃i·Ãi) · A
♦ 公式分解： ΔWi = (α · B₀ + (1-α) · B̃i · Ãi) · A 其中： A ∈ Rr×k ← 所有任务共享的下投影矩阵 B₀ ∈ Rd×r ← 所有任务共享的基础上投影矩阵 B̃i ∈ Rd×r̃ ← 任务 i 特有的低秩因子 Ãi ∈ Rr̃×r ← 任务 i 特有的低秩因子 (r̃ < r) α ∈ (0,1) ← 可学习融合系数，sigmoid(logit)
💡
直觉理解：把 LoRA 的 B 矩阵拆成两部分 — 一个大家共用的"底座" B₀，和一个每个任务自己的小调整 B̃i·Ãi。α 控制两者的比例。相当于在 LoRA 里面再嵌套一个 LoRA — 这就是 "LoRA in LoRA" 的名字由来。

4. 防遗忘的正则化机制

光共享参数还不够，B₀ 在学新任务时会被更新，可能破坏旧任务的表示。论文引入余弦正则化：

♦ 正则化损失： Lreg = (1 - cos(B̃t·Ãt, B̃t-1·Ãt-1)) · ||B₀t - B₀t-1||² ♦ 总损失： Ltotal = Ltask + λ · Lreg

公式含义

  • cos(...)：新旧任务的 B̃·Ã 方向越相似，正则越小
  • ||B₀ 变化||：共享基础矩阵变化越大，惩罚越大
  • 两者相乘：只有当任务特定部分方向变了且共享部分也变了，才重罚

Ablation 证明正则化很关键

7. 核心总结

  1. CKA 发现 — 不同任务的 LoRA A 矩阵高度相似，B 矩阵差异大。所以共享 A，分解 B。
  2. "LoRA in LoRA" 结构 — B 矩阵 = 共享基础 B₀ + 任务特定的低秩分解 B̃i·Ãi，用可学习 α 融合。
  3. 余弦正则化 — 防止共享基础在学新任务时偏移过大，是防遗忘的关键。
  4. 结果 — 每任务参数 104.6 MB（vs 357.3 MB），遗忘 BWT = -3.13（vs -48.10），性能接近独立 LoRA 上界。
一句话总结：LiLoRA 通过在 LoRA 的 B 矩阵里再嵌套一层低秩分解（LoRA-in-LoRA），让多个任务共享公共参数的同时保留各自的适配能力，配合余弦正则化大幅减少遗忘，实现了参数量和遗忘率双降 70%。

LiLoRA: 从论文公式到代码实现

对照 mmit/src/mmit/training/methods/lilora.py

Step 1. 定义 LiLoRA 层 — 论文公式的直接翻译

核心思路：每个被适配的 Linear 层旁边挂一个 LiLoRALinear 模块。它存储论文中的所有矩阵参数，并在 forward() 中计算 ΔW·x。
论文公式
A ∈ Rr×k 共享下投影 B₀ ∈ Rd×r 共享基础上投影 α = sigmoid(logit) 可学习融合系数 B̃i ∈ Rd×r̃ 任务特定 Ãi ∈ Rr̃×r 任务特定 (r̃ < r)
代码：LiLoRALinear.__init__()
class LiLoRALinear(nn.Module):
  def __init__(self, in_features, out_features,
               rank=128, task_rank=64):
    # A ∈ R^{r×k} — 共享下投影
    self.shared_A = nn.Parameter(
        torch.randn(rank, in_features) * 0.01)

    # B₀ ∈ R^{d×r} — 共享基础上投影
    self.shared_B0 = nn.Parameter(
        torch.zeros(out_features, rank))

    # α = sigmoid(logit) — 可学习融合
    self._alpha_logit = nn.Parameter(
        torch.randn(1))

    # B̃ᵢ ∈ R^{d×r̃} — 任务特定
    self.B_tilde = nn.Linear(
        task_rank, out_features, bias=False)
    # Ãᵢ ∈ R^{r̃×r} — 任务特定
    self.A_tilde = nn.Linear(
        rank, task_rank, bias=False)
论文前向传播
Step a: Ax = A · x [B, seq, r] Step b: 共享路径 = B₀ · Ax [B, seq, d] Step c: 任务路径 = B̃i · Ãi · Ax [B, seq, d] Step d: δ = α·(b) + (1-α)·(c) Output: W·x + δ
代码：LiLoRALinear.forward()
def forward(self, x, base_output):
  # Step a: 共享下投影 A·x
  Ax = F.linear(x, self.shared_A)

  # α = sigmoid(logit)
  alpha = torch.sigmoid(self._alpha_logit)

  # Step c: 任务路径 B̃·Ã·Ax
  task_BA = self.B_tilde(self.A_tilde(Ax))

  # Step d: 加权融合
  delta = (alpha * F.linear(Ax, self.shared_B0)
         + (1 - alpha) * task_BA)

  # Output: 原始输出 + delta
  return base_output + delta
🔍
初始化策略的含义：
• shared_A：小随机值 (* 0.01)，让初始 delta 接近零
• shared_B0：全零初始化 → 训练开始时 δ ≈ 0，不破坏预训练权重
• B_tilde：全零 / A_tilde：小随机 → 任务路径也从零开始
• _alpha_logit：随机初始化 → sigmoid 后 α ≈ 0.5，两条路径各占一半

Step 2. 注入模型 — 用 Hook 把 LiLoRA 挂到每一层

论文描述
对 VLM 中每个目标 Linear 层: 原始: y = W · x 改为: y = W · x + δ(x) 目标模块：q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj (即 Attention + FFN 的所有线性层) 冻结原模型所有参数 只训练 LiLoRA 新增参数
代码：LiLoRAMethod.prepare_model()
def prepare_model(self, model, processor, config):
  rank = config.get("shared_rank", 128)
  task_rank = config.get("task_rank", 64)

  # 1. 冻结整个模型
  for p in model.parameters():
    p.requires_grad = False

  # 2. 自动检测目标模块名
  targets = _auto_detect_targets(model)
  # → {"q_proj","k_proj","v_proj",...}

  # 3. 遍历所有 Linear，匹配则注入
  for name, module in model.named_modules():
    if isinstance(module, nn.Linear):
      short = name.split(".")[-1]
      if short in targets:
        # 创建 LiLoRA 模块
        ll = LiLoRALinear(
          module.in_features,
          module.out_features,
          rank, task_rank)

        # 4. 用 forward hook 注入
        def make_hook(ll):
          def hook(mod, inp, out):
            return ll(inp[0], out)
          return hook
        module.register_forward_hook(
          make_hook(ll))
💡
Forward Hook 机制（关键技巧）：
PyTorch 的 register_forward_hook(fn) 会在模块的 forward() 执行完后自动调用 fn(module, input, output)。
所以 hook(mod, inp, out) = ll(inp[0], out) 意思是：
• inp[0] = 原始输入 x → 传给 LiLoRA 计算 δ(x)
• out = 原始输出 W·x → LiLoRA 在上面加 δ
• 最终返回 W·x + δ — 完全不修改原模型代码！

Step 3. 训练循环 — 只更新 LiLoRA 参数

论文：训练策略
Loss = 标准交叉熵 (论文中正则化用于多任务 持续学习场景) Optimizer: 只优化 LiLoRA 新增的参数： - shared_A, shared_B0 - _alpha_logit - B_tilde.weight - A_tilde.weight 冻结：原模型所有参数 梯度：只从 LiLoRA 模块回传
代码：compute_loss() + get_trainable_params()
# ── 损失函数 ──
def compute_loss(self, model, batch, outputs):
  # 标准交叉熵，和普通 LoRA 一样
  return _standard_ce_loss(outputs, batch), {}

# ── 收集可训练参数 ──
def get_trainable_params(self, model):
  params = []
  for ll in self._lilora_modules:
    params.extend(
      p for p in ll.parameters()
        if p.requires_grad)
  return [{"params": params}]

# __main__.py 中的训练循环：
optimizer = AdamW(
  method.get_trainable_params(model),
  lr=2e-4)

for batch in dataloader:
  outputs = model(**batch)
  loss, _ = method.compute_loss(
    model, batch, outputs)
  loss.backward()  # 梯度只流向 LiLoRA
  optimizer.step()

Step 4. 保存和加载 — 只存 LiLoRA 状态

论文思路
保存： 只需保存每层 LiLoRA 的参数： - shared_A, shared_B0 - _alpha_logit - B_tilde, A_tilde (基座模型不变，不需要保存) 加载： 1. 加载原始基座模型 2. 重新 prepare_model() → 创建空 LiLoRA 模块并挂 hook 3. 从文件加载权重到 LiLoRA 模块 4. 模型即可推理
代码：save_checkpoint() + load_for_inference()
# ── 保存 ──
def save_checkpoint(self, model, proc, path, meta):
  lilora_state = {}
  for i, ll in enumerate(self._lilora_modules):
    lilora_state[f"layer_{i}"] = ll.state_dict()
  torch.save(lilora_state,
    os.path.join(path, "lilora_state.pt"))

# ── 加载 ──
def load_for_inference(self, path, base_model_id):
  # 1. 加载原始基座模型
  model = AutoVLM.from_pretrained(base_model_id)

  # 2. 重建 LiLoRA 结构（空参数）
  self.prepare_model(model, processor, config)

  # 3. 加载保存的权重
  state = torch.load("lilora_state.pt")
  for i, ll in enumerate(self._lilora_modules):
    ll.load_state_dict(state[f"layer_{i}"])

  model.eval()
  return model, processor, info

Step 5. 完整数据流总览

▶ 输入 x 经过模型的一个 Linear 层 (例如 q_proj) 的完整流程： x: [batch, seq_len, d_in] 输入 hidden state | ┌───────┴───────┐ │ │ 原始路径 LiLoRA 路径 (hook) W · x | = base_output shared_A · x → Ax [batch, seq, r=128] │ | │ ┌────┴────┐ │ │ │ │ B₀ · Ax Ã·Ax → [batch, seq, r̃=64] │ [共享] B̃·(Ã·Ax) → [batch, seq, d_out] │ │ [任务特定] │ └────┬────┘ │ | │ δ = α·(B₀·Ax) + (1-α)·(B̃·Ã·Ax) │ | └───────┬───────┘ | output = base_output + δ → [batch, seq, d_out]

从论文到代码的核心模式：
1. 公式 → Module：论文的每个数学公式对应一个 nn.Module，参数用 nn.Parameter 声明
2. 矩阵乘法 → F.linear()：所有 W·x 变成 F.linear(x, W) 或 nn.Linear
3. 架构修改 → Hook：不改原模型，用 register_forward_hook 在输出上 "加一笔"
4. 冻结 → requires_grad=False：一行代码冻结原始参数
5. 保存/加载 → state_dict()：只存新增参数，加载时重建结构再填入权重

B̃/Ã 的"不同"到底指什么？

三个维度：Transformer 层 × 层内模块 × 任务

先搞清楚：一个模型里有多少个 Linear？

以 Qwen2.5-VL-3B (36 层) 为例： 每一层 (Layer) 有 7 个 Linear 模块 (Module)： Layer 0: q_proj k_proj v_proj o_proj gate_proj up_proj down_proj Layer 1: q_proj k_proj v_proj o_proj gate_proj up_proj down_proj Layer 2: q_proj k_proj v_proj o_proj gate_proj up_proj down_proj ... Layer 35: q_proj k_proj v_proj o_proj gate_proj up_proj down_proj 总计: 36 层 × 7 模块 = 252 个 Linear 每个 Linear 都是一个独立的 W 矩阵，彼此没有关系。

论文 vs 我们的代码：参数共享方式不同

论文设计（持续多任务学习）
三个维度同时存在: 跨层共享： 所有 252 个 Linear → 同一个 A 所有 252 个 Linear → 同一个 B₀ 跨任务不同： 任务1 → 自己的 B̃₁, Ã₁ 任务2 → 自己的 B̃₂, Ã₂ 任务3 → 自己的 B̃₃, Ã₃ "共享" = 跨所有层 + 跨所有任务 "任务特定" = 每个任务各一份
我们的代码（单任务简化版）
只有一个维度：每个 Linear 独立 每个 Linear 有自己完整的一套： Layer0.q_proj → A₁, B0₁, B̃₁, Ã₁, α₁ Layer0.k_proj → A₂, B0₂, B̃₂, Ã₂, α₂ Layer0.v_proj → A₃, B0₃, B̃₃, Ã₃, α₃ ... Layer35.down → A₂₅₂, B0₂₅₂, ... 252 个 LiLoRALinear，互不共享。 没有多任务，只有一个训练场景。
💡
为什么我们的代码不完全按论文来？
论文的核心贡献是持续学习（sequential multi-task），共享参数 + 正则化是为了防遗忘。
我们的场景是单任务微调（一个数据集训到底），没有"旧任务"要保护，所以：
• 不需要跨层共享 A（各层输入维度可能不同）
• 不需要正则化（没有旧任务的 B̃ 要对齐）
• 保留了"B 矩阵分解为 B₀ + B̃·Ã"的结构（这本身就是一种参数高效设计）

论文中的完整图景

▶ 持续学习过程（论文场景）： 初始化: 创建共享参数 A, B₀（随机初始化） 学习任务 1 (ScienceQA): 创建 B̃₁, Ã₁（任务 1 专用） 训练时更新: A, B₀, B̃₁, Ã₁, α ← 全部参数一起训练 训练完 → 冻结 B̃₁, Ã₁（保存下来） 学习任务 2 (TextVQA): 创建 B̃₂, Ã₂（任务 2 专用，新的参数） 训练时更新: A, B₀, B̃₂, Ã₂, α ← 共享参数继续更新！ L_reg 正则化: 确保 B₀ 不偏离太远，保护任务 1 的表现 训练完 → 冻结 B̃₂, Ã₂ 学习任务 3 (Flickr30k): 创建 B̃₃, Ã₃ ...同理 推理时: 任务 1 → 加载 A, B₀ + B̃₁, Ã₁ 任务 2 → 加载 A, B₀ + B̃₂, Ã₂ 任务 3 → 加载 A, B₀ + B̃₃, Ã₃ ↑ 同一份 ↑ 各自的
一句话总结：
论文中 B̃/Ã 的"不同"指的是不同训练任务（不同数据集），不是不同的 Transformer 层。
A 和 B₀ 跨所有层 + 所有任务共享——这是论文参数量减少 70% 的关键。
我们的单任务实现做了简化：每个 Linear 独立一套参数，没有跨层共享，也没有多任务。
