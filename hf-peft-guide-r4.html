peft · 第四轮：读懂 mmit 的训练代码

前三轮你懂了 LoRA 和 peft。这一轮看 mmit 怎么用它们 — 逐行读代码，判断对不对。

mmit 的训练系统长什么样？

mmit 的训练系统设计思路是：把"怎么训"拆成可插拔的零件。你不需要改训练循环代码，只需要换零件。

四个核心角色：

① Trainer（trainer.py）— 训练循环的指挥官。它自己不懂 LoRA，但会调用下面三个零件
② PEFTStrategy（peft_strategy.py）— "怎么改模型"。决定冻结什么、注入什么、保存什么
③ LossFunction（loss.py）— "怎么算 loss"。可以换不同的损失函数
④ Config（schema.py）— "参数从哪来"。所有超参数的定义
类比理解：
Trainer 是厨房经理 — 它负责整个做菜流程（备料→烹饪→装盘）。
PEFTStrategy 是厨师 — 告诉经理"这道菜怎么做"。
LossFunction 是品控 — 告诉经理"做得好不好"。
Config 是菜谱 — 记录所有参数。

想换一种训练方法？换厨师（Strategy）就行，厨房经理和流程不用改。
涉及的文件：
mmit/training/peft_strategy.py — PEFTStrategy 基类 + LoRAStrategy 实现
mmit/training/trainer.py — Trainer 训练循环
mmit/config/schema.py — PEFTConfig, TrainingConfig 配置
mmit/examples/lora_in_lora_strategy.py — LoRA-in-LoRA 高级示例

PEFTStrategy — 四个必须实现的方法

PEFTStrategy 是一个抽象基类（ABC）。它定义了一个"合同"：任何训练策略都必须实现这四个方法。

class PEFTStrategy(ABC): # 必须实现的 4 个方法： def wrap_model(self, method, config): # ① 注入：改模型 ... def trainable_params(self, method): # ② 告知：哪些参数要训练 ... def save_adapter(self, method, path): # ③ 保存：存 adapter ... def load_adapter(self, method, path): # ④ 加载：读 adapter ...
每个方法的职责：

wrap_model(method, config)
→ 冻结不该训的参数，注入该训的东西（比如 LoRA）
→ 第二轮你学的 get_peft_model 就是在这里调用的

trainable_params(method)
→ 返回 [{"params": [...], "lr": 2e-5}] 给 optimizer
→ 告诉训练循环"只训这些参数"

save_adapter(method, path)
→ 只保存 adapter 权重（不保存完整模型）
→ 第二轮你学的 model.save_pretrained 就是在这里调用的

load_adapter(method, path)
→ 从文件加载 adapter 权重回模型
为什么参数是 method 而不是 model？
因为 mmit 是多模态框架。一个 Method 对象里有三个子模块：
method.vision — 视觉编码器（比如 CLIP）
method.projector — 投影层（把图像特征映射到语言空间）
method.llm — 语言模型（比如 Llama）

传 method 进来，让策略可以自由选择改哪个部分。
比如 LoRA 只改 llm，而 projector_only 只改 projector。
还有 2 个可选方法（持续学习用）：
on_task_start(method, task_id, config) — 新任务开始前调用
on_task_end(method, task_id, config) — 一个任务训完后调用
默认是空的（no-op）。只有做持续学习的策略才需要实现它们。

LoRAStrategy.wrap_model() — 逐行阅读

LoRAStrategy 是 mmit 内置的 LoRA 实现。它的 wrap_model 就是调用你在第二轮学过的 peft API：

def wrap_model(self, method, config): from peft import LoraConfig, get_peft_model # 第一步：找到 LLM llm = self._get_llm(method) # 从 method 里取出 llm # 第二步：从 mmit 的 config 构建 peft 的 LoraConfig target = config.target_modules or ["q_proj", "v_proj"] lora_cfg = LoraConfig( r=config.rank, # 从 mmit config 读 rank lora_alpha=config.alpha, # 从 mmit config 读 alpha lora_dropout=config.dropout, # 从 mmit config 读 dropout target_modules=target, bias="none", task_type="CAUSAL_LM", ) # 第三步：调用 peft 注入 LoRA（第二轮学的！） llm = get_peft_model(llm, lora_cfg) # 第四步：把改过的 llm 放回 method self._set_llm(method, llm) # 第五步（可选）：如果 config 说要训 projector，就解冻它 if config.extra.get("train_projector", False): for p in method.projector.parameters(): p.requires_grad = True
对应到你学过的知识：
第一步 — mmit 的 method 有多个子模块，先找到 llm
第二步 — 把 mmit 的 PEFTConfig 翻译成 peft 的 LoraConfig（参数搬运）
第三步 — 你在第二轮学的核心操作：get_peft_model！
第四步 — peft 返回了一个新的 PeftModel，要放回去
第五步 — projector 不走 peft，直接用 requires_grad = True 解冻
这段代码对不对？检查清单：
✓ LoraConfig 的参数和第二轮学的一致（r, alpha, target_modules, dropout）
✓ bias="none" — 标准做法，不训练偏置
✓ task_type="CAUSAL_LM" — 因为 llm 是自回归语言模型
✓ get_peft_model 返回新模型后用 _set_llm 放回 — 正确，因为 peft 返回的是 PeftModel 包装
✓ projector 直接解冻 — 正确，projector 是小模块，不需要 LoRA

trainable_params() — 告诉 optimizer 训练什么

Trainer 创建 optimizer 时需要知道：训练哪些参数？学习率是多少？这就是 trainable_params 的职责。

def trainable_params(self, method): groups = [] # 第一组：LLM 的 LoRA 参数 llm = self._get_llm(method) llm_params = [p for p in llm.parameters() if p.requires_grad] if llm_params: groups.append({"params": llm_params}) # 用默认 lr # 第二组：Projector 参数（如果被解冻了） proj = getattr(method, "projector", None) if proj is not None: proj_params = [p for p in proj.parameters() if p.requires_grad] if proj_params: groups.append({"params": proj_params, "lr": 1e-3}) # 独立 lr return groups

返回的是一个列表，每个元素是一个 param_group 字典。这是 PyTorch optimizer 的标准格式。

为什么分两组？
LoRA 参数（A、B 矩阵）用小学习率（2e-5，Trainer 的默认值）
Projector 参数用大学习率（1e-3，写死在代码里）

原因：LoRA 是在预训练模型上做微小调整，学习率要小。
Projector 是从零训练（或者从一个简单初始化开始），学习率要大。
这段代码对不对？检查：
✓ 用 p.requires_grad 过滤 — 只拿可训练的参数（第二轮讲过，peft 冻结了 W）
✓ 两个 param_group 用不同 lr — PyTorch optimizer 支持这个，是标准做法
✓ projector 的 lr=1e-3 比 LoRA 的 lr=2e-5 大 50 倍 — 合理
✓ 如果 projector 没被解冻，proj_params 为空，不会加进去 — 正确
Trainer 怎么用这个返回值：
param_groups = self.peft.trainable_params(method)
optimizer = AdamW(param_groups)

就是直接传给 optimizer。PyTorch 的 AdamW 接受 param_group 列表。

save_adapter / load_adapter — 保存和加载

训练完成后保存 adapter，部署时加载回来。两个方法：

# ── 保存 ── def save_adapter(self, method, path): os.makedirs(path, exist_ok=True) llm = self._get_llm(method) llm.save_pretrained(path) # ← 第二轮学的！PeftModel 的方法 # 如果 projector 被训练了，也保存它 proj = getattr(method, "projector", None) if proj is not None: if any(p.requires_grad for p in proj.parameters()): torch.save(proj.state_dict(), os.path.join(path, "projector.pt")) # ── 加载 ── def load_adapter(self, method, path): from peft import PeftModel llm = self._get_llm(method) llm = PeftModel.from_pretrained(llm, path) # ← 第二轮学的！注入 + 加载权重 self._set_llm(method, llm) # 如果有 projector 权重文件，也加载 proj_path = os.path.join(path, "projector.pt") if os.path.exists(proj_path): method.projector.load_state_dict(torch.load(proj_path))
对应到第二轮的知识：
llm.save_pretrained(path) — 第二轮第 7 步学的。保存 adapter_model.safetensors + adapter_config.json
PeftModel.from_pretrained(llm, path) — 第二轮第 8 步学的。读配置 → 注入空 LoRA → 加载权重
这段代码对不对？检查：
✓ LLM 部分用 peft 的 save/load — 正确，因为 wrap_model 用了 get_peft_model
✓ Projector 用 PyTorch 原生的 state_dict — 正确，projector 没有用 peft，是普通模块
✓ 加载 projector 时先检查文件是否存在 — 正确，因为不一定训了 projector
✓ load 时要 _set_llm 放回去 — 正确，和 wrap_model 同理
保存后的文件结构：
my-adapter/
├── adapter_model.safetensors — LoRA 的 A/B 权重（~20MB）
├── adapter_config.json — LoraConfig 的 JSON 序列化
└── projector.pt — projector 权重（如果训了的话）

Trainer.train() — 训练循环的全流程

Trainer 是"指挥官"。它不懂 LoRA 的任何细节，只按顺序调用 strategy 的方法：

def train(self, dataset): # 阶段 1：让 strategy 改模型 self.peft.wrap_model(method, config.peft) # 阶段 2：让 strategy 告诉我训哪些参数 param_groups = self.peft.trainable_params(method) optimizer = AdamW(param_groups) # 阶段 3：创建 DataLoader + 学习率调度器 loader = DataLoader(dataset, batch_size=..., shuffle=True) scheduler = cosine_schedule(optimizer, warmup, total_steps) # 阶段 4：训练循环 for epoch in range(num_epochs): for batch in loader: output = method.forward_loss(batch) # forward loss = self.loss_fn.compute(output, ...) # 算 loss loss.backward() # backward optimizer.step() # 更新参数 # 每隔 N 步保存 checkpoint if global_step % save_steps == 0: self.peft.save_adapter(method, ckpt_path) # 阶段 5：训练结束，保存最终 adapter self.peft.save_adapter(method, "final")
Trainer 调用了 strategy 的哪些方法：
wrap_model — 阶段 1，训练开始前
trainable_params — 阶段 2，创建 optimizer 时
save_adapter — 阶段 4/5，中间 checkpoint + 最终保存
这就是"可插拔"的意思：
Trainer 不知道你用的是 LoRA、Prefix-tuning 还是别的方法。
它只知道：调 wrap_model（你去改模型）→ 调 trainable_params（你告诉我训什么）→ 训练 → 调 save_adapter（你去保存）。

想换成别的方法？写一个新的 PEFTStrategy 子类就行，Trainer 代码一行不改。
一些细节（代码里有，这里简化了）：
• gradient accumulation — 多步梯度累积后再 optimizer.step
• gradient clipping — clip_grad_norm_ 防止梯度爆炸
• cosine schedule — 余弦学习率调度 + warmup
• step_callback — 外部回调（比如 dashboard 更新进度条）

Config — 参数从哪来，流到哪去

mmit 的配置用 @dataclass 定义。两个和训练相关的：

# PEFTConfig — 给 strategy 用 @dataclass class PEFTConfig: name: str = "lora" # 用哪个 strategy（从注册表找） target_modules: List[str] = None # 给哪些层加 LoRA rank: int = 16 # r（你在第一轮学的 rank） alpha: int = 32 # lora_alpha（第二轮的"音量旋钮"） dropout: float = 0.05 # lora_dropout extra: Dict = {} # 策略专属参数（比如 inner_rank） # TrainingConfig — 给 Trainer 用 @dataclass class TrainingConfig: peft: PEFTConfig = PEFTConfig() # 嵌套的 PEFT 配置 loss: str = "causal_lm" # 损失函数 learning_rate: float = 2e-5 # 默认学习率 num_epochs: int = 3 per_device_batch_size: int = 4 gradient_accumulation_steps: int = 4 warmup_ratio: float = 0.03 save_steps: int = 500 ...
参数的流向：
YAML 文件 → TrainingConfig → Trainer 读取训练参数
TrainingConfig.peft → PEFTConfig → Strategy 读取 LoRA 参数

具体来说：
config.peft.rank → Strategy 里变成 LoraConfig(r=config.rank)
config.peft.alpha → 变成 LoraConfig(lora_alpha=config.alpha)
config.learning_rate → Trainer 里变成 AdamW(..., lr=config.learning_rate)
extra 字典的用途：
框架定义的参数（rank, alpha, dropout）有专门的字段。
但每个策略可能有自己独特的参数。比如 LoRA-in-LoRA 需要 inner_rank。

这些放在 extra 字典里：config.extra["inner_rank"]
框架不需要懂它们，由策略自己解析。

注册表 — Trainer 怎么找到 strategy

你在 config 里写 name="lora"，Trainer 怎么知道用 LoRAStrategy？答案是注册表（registry）。

# 在某个地方，LoRAStrategy 被注册了： from mmit import registry registry.register("peft", "lora", LoRAStrategy) # Trainer.from_config 内部： def from_config(cls, method, config): # config.peft.name = "lora" peft_strategy = registry.build("peft", config.peft.name) # → registry 里找 "peft" 类别下 "lora" 这个名字 # → 找到 LoRAStrategy 类 # → 调用 LoRAStrategy() 创建实例 # → 返回 loss_fn = registry.build("loss", config.loss) return cls(method, peft_strategy, loss_fn, config)
注册表就是一个字典的字典：
registry = {
"peft": {"lora": LoRAStrategy, "lora-in-lora": LoRAInLoRAStrategy, ...},
"loss": {"causal_lm": CausalLMLoss, ...},
"method": {"llava-v1.5": LLaVAv15Method, ...},
}

register("peft", "lora", LoRAStrategy) — 把类注册到字典里
build("peft", "lora") — 从字典里取出类，创建实例
这个设计的好处：
你写了一个新的策略类（比如 MyMoReSStrategy），只需要：

1. 继承 PEFTStrategy，实现四个方法
2. registry.register("peft", "mores", MyMoReSStrategy)
3. YAML 里写 name: mores

不需要改 Trainer 的任何代码。

LoRA-in-LoRA — 读懂一个高级策略

现在用你学到的知识，来读一个复杂的例子：LoRAInLoRAStrategy。它不用 peft 库，而是手写了双层 LoRA。

它的思路（持续学习）：
每个目标层有两个 LoRA：
y = W·x + B_outer·A_outer·x + B_inner·A_inner·x

外层（outer）：长期记忆。积累所有任务的知识，学习率小
内层（inner）：工作记忆。只学当前任务，学习率大

每个任务训完后：inner 合并进 outer → inner 重置为零 → 开始下一个任务
# LoRALinear — 手写的双层 LoRA 模块（不用 peft 库） class LoRALinear(nn.Module): def __init__(self, original, rank, alpha): self.original = original # 原始 nn.Linear（冻结） self.lora_A_outer = nn.Linear(in_d, rank) # 外层 A self.lora_B_outer = nn.Linear(rank, out_d) # 外层 B（初始为零） self.lora_A_inner = nn.Linear(in_d, rank) # 内层 A self.lora_B_inner = nn.Linear(rank, out_d) # 内层 B（初始为零） def forward(self, x): base = self.original(x) # W·x outer = self.lora_B_outer(self.lora_A_outer(x)) * scaling inner = self.lora_B_inner(self.lora_A_inner(x)) * scaling return base + outer + inner # 三路相加
验证这段代码 — 用你学过的知识：
✓ self.original 被冻结，相当于 peft 的 base_layer — 第三轮学的
✓ B 初始化为零（nn.init.zeros_）— 第三轮第 7 步学的，保证训练开始时输出不变
✓ A 用 kaiming 初始化 — 第三轮第 7 步学的，保证梯度流动
✓ forward 里三路相加 — 和 peft 的 result += B(A(x)) * scaling 原理一样
✓ * scaling（alpha/r）— 第二轮学的"音量旋钮"
和 peft 的 LoRAStrategy 的区别：
LoRAStrategy 调用 peft 库的 get_peft_model — 自动处理注入、保存、加载
LoRAInLoRAStrategy 手写了模块替换 — 因为 peft 不支持双层 LoRA

手写时，它用了第三轮学的技巧：
name.endswith(target) — 字符串匹配找目标层
setattr(parent, child_name, lora_module) — 替换 parent 的子模块
和 peft 内部的 parent._modules[name] = new_module 本质上是同一件事。

持续学习 hook + 第四轮总结

LoRA-in-LoRA 的核心在于两个 hook —— 它们让 Trainer 在任务之间自动执行"合并→重置"：

def on_task_start(self, method, task_id, config): if task_id == 0: return # 第一个任务不用动，wrap_model 已经初始化好 # 后续任务：重置 inner → B=0，准备学新东西 for m in self._adapted_modules: m.reinit_inner() # A=kaiming, B=0 def on_task_end(self, method, task_id, config): # 任务结束：inner 的知识合并进 outer，然后重置 inner for m in self._adapted_modules: m.merge_inner_to_outer() # SVD 合并 + inner 归零
Trainer 调用顺序（train_continual）：
for task_id, dataset in enumerate(tasks):
peft.on_task_start(method, task_id, config) ← inner 重置为零
trainer.train(dataset) ← 训练当前任务
peft.on_task_end(method, task_id, config) ← inner 合并进 outer

Trainer 不知道"合并"和"重置"的细节，它只按合同调 hook。
第四轮总结 — 你现在能看懂的代码：

① PEFTStrategy 抽象基类 — 4 个必须方法 + 2 个可选 hook
② LoRAStrategy.wrap_model — 怎么把 mmit config 翻译成 peft 调用
③ trainable_params — 怎么分 param_group 给 optimizer
④ save/load_adapter — LoRA 用 peft 的 save/load，projector 用 PyTorch 原生
⑤ Trainer.train — 五个阶段，调用 strategy 的方法
⑥ Config 数据流 — PEFTConfig → LoraConfig 的参数搬运
⑦ 注册表 — 名字到类的映射，实现可插拔
⑧ LoRA-in-LoRA — 手写双层 LoRA + 持续学习 hook
现在你有能力判断代码对不对：
• 看到 get_peft_model → 你知道它会冻结+注入+包装（第二轮/第三轮）
• 看到 B=0 初始化 → 你知道这是保证训练开始时模型不变（第三轮）
• 看到 requires_grad 过滤 → 你知道只训 A/B（第二轮）
• 看到 parent._modules[name] = new 或 setattr → 你知道这是模块替换（第三轮）
• 看到 save_pretrained / from_pretrained → 你知道存的是什么文件（第二轮）
