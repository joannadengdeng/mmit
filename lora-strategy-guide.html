LoRAStrategy 代码逐行讲解

像给 10 岁小孩一样，一步一步把每行代码讲明白

先看全景：LoRAStrategy 到底在干什么？

类比：想象你是一个装修工人。房子（模型）已经建好了，你不能拆房子重建，但你可以：

1. 决定在哪面墙上贴壁纸（_get_llm — 找到要改的墙）
2. 把壁纸贴上去（wrap_model — 注入 LoRA）
3. 告诉油漆工只刷壁纸区域（trainable_params — 告诉优化器只训练 LoRA 部分）
4. 拍照存档（save_adapter — 保存 LoRA 权重）
5. 下次再贴回去（load_adapter — 加载 LoRA 权重）
总共 6 个方法：2 个辅助方法（找 LLM、设 LLM）+ 4 个核心方法（贴壁纸、告诉训练谁、保存、加载）。接下来我们一个一个看。

第 1 步：_get_llm — "找到 LLM 在哪"

类比：不同品牌的电脑，电池可能在不同位置 — 有的在底部，有的在侧面。_get_llm 就是"找到电池在哪"的工具。不管电池装在哪个位置，它都能找到。
peft_strategy.py def _get_llm(self, method): for attr in ("llm", "_llm", "model"): if hasattr(method, attr): return getattr(method, attr) raise AttributeError("Cannot find LLM...")

逐行翻译

第 1 行：for attr in ("llm", "_llm", "model") — 按顺序猜三个名字

第 2 行：hasattr(method, attr) — 问 method："你有叫这个名字的东西吗？"

第 3 行：getattr(method, attr) — "有的话拿出来给我"

第 4 行：三个名字都没找到？报错："我找不到你的 LLM！"

为什么要猜三个名字？因为不同的研究员给 LLM 取的属性名不同：
• LLaVA 用 method.llm
• 有些模型用 method._llm（下划线表示"私有"）
• 还有些用 method.model
这个函数兼容所有情况。

小测验：如果一个 Method 有 .model 属性但没有 .llm，_get_llm 会怎样？

对！它会依次检查 "llm" → 没有 → "_llm" → 没有 → "model" → 有！返回 method.model。

第 2 步：_set_llm — "把新 LLM 放回去"

类比：_get_llm 是"取出电池"，_set_llm 是"装回电池"。你取出了旧电池，改装了一下（加了 LoRA），然后要装回原来的位置。
peft_strategy.py def _set_llm(self, method, llm): for attr in ("llm", "_llm", "model"): if hasattr(method, attr): setattr(method, attr, llm) return

逐行翻译

setattr(method, attr, llm) — 意思是 method.llm = llm，但用的是变量名。

和 _get_llm 一样的逻辑：找到原来那个属性名，然后把新的 LLM 设置回去。

getattr vs setattr：
• getattr(method, "llm") 等于 method.llm — 读取
• setattr(method, "llm", new_llm) 等于 method.llm = new_llm — 写入

为什么不直接写 method.llm = llm？因为属性名是变量（可能是 "llm"、"_llm" 或 "model"），所以必须用 setattr。

第 3 步：wrap_model — 核心！注入 LoRA

类比：这是整个装修的核心操作。你要：
1. 先找到要改的墙（LLM）
2. 选好要贴壁纸的位置（target_modules）
3. 配好壁纸的参数（颜色、厚度 → rank、alpha）
4. 让专业贴壁纸的公司来贴（HuggingFace peft 库）
5. 把贴好壁纸的墙装回去
wrap_model — 上半部分：LoRA 注入 def wrap_model(self, method, config): from peft import LoraConfig, get_peft_model # ❶ 导入工具 llm = self._get_llm(method) # ❷ 找到 LLM target = config.target_modules or ["q_proj", "v_proj"] # ❸ 选位置 lora_cfg = LoraConfig( # ❹ 配参数 r=config.rank, # 低秩维度 lora_alpha=config.alpha, # 缩放因子 lora_dropout=config.dropout, # 防过拟合 target_modules=target, # 贴哪些层 bias="none", # 不改 bias task_type="CAUSAL_LM", # 文本生成任务 ) llm = get_peft_model(llm, lora_cfg) # ❺ 动手贴！ self._set_llm(method, llm) # ❻ 装回去

❶ from peft import ...

导入 HuggingFace 的 peft 库。LoraConfig 是 LoRA 的"配方单"，get_peft_model 是"施工队"。

❷ llm = self._get_llm(method)

用上一步学的方法，从 Method 里找到 LLM。

❸ target = config.target_modules or ["q_proj", "v_proj"]

决定要在 LLM 的哪些层注入 LoRA。如果配置里没写，就默认用 q_proj 和 v_proj（这两个是 attention 里的 Query 和 Value 矩阵）。

or 的意思：A or B = 如果 A 有值就用 A，否则用 B。
所以 config.target_modules or ["q_proj", "v_proj"] 意思是：用户指定了就用用户的，没指定就用默认值。

❹ LoraConfig(...) — 写"配方单"

把所有参数打包成一个配置对象，告诉 peft 库你要什么样的 LoRA。

❺ get_peft_model(llm, lora_cfg) — 最关键的一行！

这一行调用 HuggingFace 的 peft 库，它会自动做这些事：

1. 遍历 LLM 的所有层，找到名字匹配 target_modules 的 nn.Linear
2. 在每个匹配的层旁边加一对小矩阵 A 和 B（就是 LoRA）
3. 冻结原始权重（requires_grad = False）
4. 返回一个"包装后的 LLM"

❻ self._set_llm(method, llm)

把包装后的 LLM 放回 Method 里，替换原来的 LLM。

第 4 步：wrap_model 的"可选项" — 要不要同时训练 Projector？

wrap_model — 下半部分：可选 Projector # Optionally unfreeze projector if config.extra.get("train_projector", False): proj = getattr(method, "projector", None) if proj is not None: for p in proj.parameters(): p.requires_grad = True

逐行翻译

config.extra.get("train_projector", False)

去 config 的 extra 字典里找 "train_projector" 这个键。找到了就用它的值，找不到就用 False。

类比：extra 就像一个"备注栏"。框架定义了一些标准字段（rank、alpha 等），但研究员可能有额外需求。extra 让他们自由加参数，框架不管里面有什么。

就像点外卖时的"备注"栏 — 店家不会预设你要写什么，但你可以写"不要辣"、"多加葱"之类的。

getattr(method, "projector", None)

尝试获取 method.projector。如果 method 没有 projector 属性，就返回 None（而不是报错）。

p.requires_grad = True

解冻 projector 的每个参数，让它们可以被训练。默认情况下 get_peft_model 会冻结所有非 LoRA 参数，所以这里需要手动解冻。

小测验：如果用户的配置是 PEFTConfig(name="lora", rank=16)，没有设 extra，Projector 会被训练吗？

对！config.extra 默认是空字典 {}。调用 {}.get("train_projector", False) 找不到键，返回默认值 False。所以整个 if 块被跳过，Projector 保持冻结。

第 5 步：trainable_params — "告诉优化器该训练谁"

类比：装修完了，你给油漆工一份清单："这些地方需要上色（训练），其他地方别碰（冻结）"。而且不同区域可能用不同颜色（学习率）。
trainable_params def trainable_params(self, method): groups = [] # 空清单 # LLM LoRA params llm = self._get_llm(method) llm_params = [p for p in llm.parameters() if p.requires_grad] # 只要"解冻"的参数 if llm_params: groups.append({"params": llm_params}) # 加入清单 # Projector params (if unfrozen) proj = getattr(method, "projector", None) if proj is not None: proj_params = [p for p in proj.parameters() if p.requires_grad] if proj_params: groups.append({"params": proj_params, "lr": 1e-3}) # Projector 用大 lr！ return groups

关键概念：param_groups

PyTorch 的优化器可以接收"参数组"列表。每个组是一个字典：

组 1 — LLM LoRA：{"params": [A矩阵们, B矩阵们]} — 用 Trainer 默认 lr（如 2e-5）
组 2 — Projector：{"params": [proj参数们], "lr": 1e-3} — 用更大的 lr

为什么 Projector 用大 lr？因为 Projector 参数量小，而且在 Stage 1 时可能是从零学的，需要更快的学习速度。LoRA 参数加在已有知识的 LLM 上，所以用小 lr 微调就够了。

if p.requires_grad 是什么意思？只收集"解冻的"参数。get_peft_model 已经帮我们冻结了原始权重，只有 LoRA 的 A、B 矩阵是 requires_grad=True 的。所以这行代码自动过滤出 LoRA 参数。

第 6 步：save_adapter — "只保存壁纸，不保存整面墙"

类比：你装修了一面墙，现在要搬家。你不需要把整面墙拆走（14GB），只需要把壁纸揭下来带走（20MB），到新家再贴上去就行。
save_adapter def save_adapter(self, method, path): os.makedirs(path, exist_ok=True) # 创建保存目录 llm = self._get_llm(method) llm.save_pretrained(path) # HF peft 自带的保存方法 # Save projector if it was trained proj = getattr(method, "projector", None) if proj is not None: if any(p.requires_grad for p in proj.parameters()): torch.save(proj.state_dict(), os.path.join(path, "projector.pt"))

逐行翻译

os.makedirs(path, exist_ok=True) — 创建目录，如果已存在就不报错。

llm.save_pretrained(path) — 这是 HuggingFace peft 库的方法！它只保存 LoRA 的 A/B 矩阵，不保存原始的大模型权重。这就是为什么文件只有 ~20MB 而不是 ~14GB。

any(p.requires_grad for p in proj.parameters()) — 检查 Projector 的参数里有没有任何一个是被训练的。如果有，就额外保存 Projector 的权重。

any() 是什么？
any([True, False, False]) → True（只要有一个 True 就行）
any([False, False, False]) → False（全是 False 才返回 False）

这里用来检测"Projector 是否被训练过"。

第 7 步：load_adapter — "把壁纸贴回去"

load_adapter def load_adapter(self, method, path): from peft import PeftModel llm = self._get_llm(method) llm = PeftModel.from_pretrained(llm, path) # 从文件加载 LoRA self._set_llm(method, llm) # 装回去 # 如果有 projector 权重也一起加载 proj_path = os.path.join(path, "projector.pt") if os.path.exists(proj_path): proj = getattr(method, "projector", None) if proj is not None: proj.load_state_dict( torch.load(proj_path, weights_only=True))

逐行翻译

PeftModel.from_pretrained(llm, path) — peft 库的方法：读取之前保存的 LoRA 权重文件，在 LLM 上重新注入 LoRA 层，并加载权重。和 save_pretrained 配对使用。

os.path.exists(proj_path) — 检查 projector.pt 文件是否存在。保存时如果 Projector 没有被训练，就不会有这个文件。

weights_only=True — 安全选项。只加载权重数据，不执行文件里可能包含的任意代码（防止恶意文件攻击）。

小测验：save 和 load 是怎么配对的？

对！LoRA 部分用 HuggingFace peft 自带的方法（它知道怎么只保存 LoRA 参数），Projector 部分用 PyTorch 原生的 torch.save/load（因为它不是 LoRA，是普通参数）。

完整串联：Trainer 怎么调用 LoRAStrategy 的？

现在你知道了每个方法的功能。让我们看看 Trainer 是怎么按顺序调用它们的：

Trainer.train() 的简化版 # 第 1 步：注入 LoRA self.peft.wrap_model(method, config.peft) # 第 2 步：获取要训练的参数，创建优化器 param_groups = self.peft.trainable_params(method) optimizer = AdamW(param_groups) # 第 3 步：训练循环 for epoch in range(num_epochs): for batch in dataloader: loss = method.forward_loss(batch) # 前向传播 loss.backward() # 反向传播（只有 LoRA 有梯度） optimizer.step() # 更新 LoRA 参数 # 第 4 步：保存 self.peft.save_adapter(method, "output/final")
关键洞察：Trainer 完全不知道 LoRA 是什么！它只调用 wrap_model、trainable_params、save_adapter 这几个接口。你可以把 LoRAStrategy 换成任何其他策略（Adapter、MoReS……），Trainer 的代码一行都不用改。

这就是策略模式的力量 — 把"变化的部分"（怎么注入适配器）从"不变的部分"（训练循环）中分离出来。

最终测验：如果你想把 LoRA 换成 Adapter Layer，需要改 Trainer 的代码吗？

完全正确！Trainer 只认 PEFTStrategy 接口，不认具体实现。你写一个 AdapterLayerStrategy 继承 PEFTStrategy，实现 4 个方法，注册到 registry，改一下配置文件就行了。Trainer、Method、数据管道 — 全都不用动。
