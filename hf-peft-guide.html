peft · 第二轮：代码怎么写

第一轮你懂了原理。这一轮：每个概念对应的代码是什么，一行一行看。

从概念到代码：只有三步

第一轮你学了：peft 做三件事 — 配置、注入、训练。对应到代码，核心也就三行：

# 第一行：写一张配置单 — "我要什么样的 LoRA" config = LoraConfig(r=16, lora_alpha=32, target_modules=["q_proj", "v_proj"]) # 第二行：把配置单交给 peft，让它注入 LoRA model = get_peft_model(model, config) # 第三行：正常训练，和没有 LoRA 的时候写法完全一样 loss.backward() # 梯度只到 A 和 B
就这三行。剩下的全是细节。
接下来每一步我们拆开看：每行代码里的参数是什么意思、运行后模型变成了什么样。

第一行：LoraConfig — 写一张配置单

LoraConfig 就是一张配置单。你在上面填："LoRA 的大小是多少、给哪些层加、缩放系数多少"。它自己不做任何操作，只是记录你的要求。

config = LoraConfig( r=16, # ← 最重要的：rank，A 和 B 的中间维度 lora_alpha=32, # ← 缩放系数（下一步详细讲） target_modules=["q_proj", "v_proj"], # ← 给哪些层加 LoRA )

只需要记住这三个参数。剩下的都有默认值，可以先不管：

三个必须理解的参数：
• r=16 — 第一轮讲过的 rank。控制 A 和 B 矩阵的大小。常用值：8、16、32
• lora_alpha=32 — 缩放系数。通常设为 r 的 1~2 倍。下一步详细讲
• target_modules=["q_proj", "v_proj"] — 哪些层要加 LoRA。后面详细讲匹配规则
其他参数（先知道有就行，不用记）：
• lora_dropout=0.05 — LoRA 旁路上加 dropout，防过拟合
• bias="none" — 是否训练偏置项。"none" 表示不训练
• task_type=TaskType.CAUSAL_LM — 告诉 peft 你用的是什么类型的模型
• modules_to_save=["embed_tokens"] — 不加 LoRA 但要解冻训练的层

lora_alpha 是什么？— LoRA 旁路的"音量旋钮"

第一轮讲了 LoRA 的计算：输出 = W·x + B·A·x。但实际上完整的公式是：

# 完整公式 输出 = W·x + (α/r) × B·A·x

α 就是 lora_alpha，r 就是 rank。α/r 是一个缩放系数，控制 LoRA 旁路对最终输出的影响有多大。

打个比方：
原始模型的输出是"主音轨"（W·x），LoRA 的输出是"伴奏"（B·A·x）。
α/r 就是伴奏的音量旋钮。

• alpha=16, r=16 → α/r = 1.0 — 伴奏和主音轨同样大声
• alpha=32, r=16 → α/r = 2.0 — 伴奏是主音轨的两倍大声
• alpha=8, r=16 → α/r = 0.5 — 伴奏只有主音轨一半声音
实际怎么选？
• 最常见的组合：r=16, alpha=32（α/r=2）或 r=16, alpha=16（α/r=1）
• 经验法则：alpha 设为 r 的 1~2 倍
• 太大：LoRA 的变化盖过原始模型，训练不稳定
• 太小：LoRA 几乎没影响，等于没学
为什么不直接调 r？
改 r 会改变 A 和 B 矩阵的大小（参数量变了），需要重新初始化。
改 alpha 只是改一个乘法系数，参数量不变。
所以 alpha 是用来"微调 LoRA 影响力"的旋钮，比改 r 方便得多。

target_modules — 告诉 peft "改哪些层"

模型是一棵模块树（第一轮讲过）。每个模块有一个名字，比如 model.layers.0.self_attn.q_proj。

peft 怎么找到要改的层？它做字符串结尾匹配：

匹配规则：
你写 target_modules=["q_proj", "v_proj"]，peft 就会遍历模型的每一个模块，
检查模块名字是不是以 "q_proj" 或 "v_proj" 结尾。

比如 "model.layers.0.self_attn.q_proj" — 以 "q_proj" 结尾 → 匹配！注入 LoRA
"model.layers.0.self_attn.k_proj" — 不匹配 → 跳过
"model.layers.15.self_attn.q_proj" — 以 "q_proj" 结尾 → 匹配！

所有 32 层的 q_proj 和 v_proj 都会被匹配到 → 总共 64 个层被注入 LoRA。
peft 内部的伪代码： for name, module in model.named_modules(): for target in target_modules: if name.endswith(target): # ← 就是字符串结尾匹配 replace_with_lora(module) # ← 替换成 lora.Linear
如果想更精确地匹配：
可以传正则表达式，比如 target_modules=".*attn.*(q|v)_proj"。
这样只匹配 attention 里的 q_proj 和 v_proj，不会意外匹配到 MLP 里同名的层。

但一般来说，简单的列表就够用了。

第二行：get_peft_model() — 一键注入

配置单写好了，现在交给 peft 执行。get_peft_model() 接收两个参数：原始模型 + 配置单。

from transformers import AutoModelForCausalLM from peft import LoraConfig, get_peft_model # 先加载原始模型 model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-hf") # 写配置单 config = LoraConfig(r=16, lora_alpha=32, target_modules=["q_proj", "v_proj"]) # 一键注入！ model = get_peft_model(model, config) # 看看注入后的结果 model.print_trainable_parameters()
trainable params: 8,388,608 || all params: 6,746,804,224 || trainable%: 0.1243

这一行调用后，模型发生了三个变化：

get_peft_model 内部做了什么：

① 冻结 — 把模型所有参数的 requires_grad 设为 False（全部不训练）

② 注入 — 遍历模块树，把匹配到的 Linear 替换成 lora.Linear
（lora.Linear 内部有原始 W + 新的 A 和 B，A/B 的 requires_grad=True）

③ 包装 — 把整个模型包进 PeftModel 外壳
（PeftModel 提供 save_pretrained、merge_and_unload 等方法）
注意：model 被覆盖了
model = get_peft_model(model, config) 这行把原来的变量名覆盖了。
现在 model 不再是普通的 LlamaForCausalLM，而是一个 PeftModel。
但你可以像以前一样用它 — model(**batch)、model.parameters() 都照常工作。

验证一下：注入前后 print(model) 的变化

用 print(model) 可以看到模型结构。对比注入前后，你能清楚看到 peft 改了什么：

注入前：普通的 LlamaForCausalLM

(self_attn): (q_proj): Linear(4096, 4096) (k_proj): Linear(4096, 4096) (v_proj): Linear(4096, 4096) (o_proj): Linear(4096, 4096)

注入后：变成了 PeftModel

(self_attn): (q_proj): lora.Linear( (base_layer): Linear(4096, 4096) ← W 还在，冻结 (lora_A): Linear(4096, 16) ← 新加的 A (lora_B): Linear(16, 4096) ← 新加的 B ) (k_proj): Linear(4096, 4096) ← 没变 (v_proj): lora.Linear( (base_layer): Linear(4096, 4096) (lora_A): Linear(4096, 16) (lora_B): Linear(16, 4096) )
对应到第一轮的概念：
• q_proj 和 v_proj 从 Linear 变成了 lora.Linear — 这就是"注入"
• lora.Linear 里面的 base_layer 就是原来的 W（冻结❄）
• lora_A 和 lora_B 就是新加的两个小矩阵（可训练🔥）
• 16 就是你设的 r=16
• k_proj 和 o_proj 没有变 — 因为你没把它们写进 target_modules

第三行：训练 — 代码和平时一模一样

LoRA 注入后，训练循环不需要任何修改。一行一行来看：

# 创建 optimizer — 和平时一样 optimizer = torch.optim.AdamW(model.parameters(), lr=2e-4) for batch in dataloader: outputs = model(**batch) # ① forward loss = outputs.loss # ② 算 loss loss.backward() # ③ backward optimizer.step() # ④ 更新参数 optimizer.zero_grad() # ⑤ 清梯度

和普通训练完全一样。但 peft 在背后偷偷做了区分：

每一行背后发生了什么：

model.parameters() — PyTorch 只返回 requires_grad=True 的参数。
　→ 冻结的 W 不会返回，只返回 A 和 B → optimizer 只管 A 和 B

model(**batch) — 每个 lora.Linear 算 W·x + (α/r)·B·A·x
　→ 两条路（原路 + 旁路）都跑，结果相加

loss.backward() — 梯度只流向 requires_grad=True 的参数
　→ W 不存梯度（省显存！），A 和 B 存梯度

optimizer.step() — 只更新 A 和 B（optimizer 只知道这些参数）
不需要手动过滤参数：
你不用写 filter(lambda p: p.requires_grad, model.parameters())。
直接传 model.parameters() 就行。PyTorch 的 optimizer 本来就会跳过
requires_grad=False 的参数（不会给它们分配梯度和状态）。

保存 — 只存 20MB 的 adapter

训练完了，保存结果。PeftModel 的 save_pretrained 只保存 LoRA 部分：

# 一行代码保存 model.save_pretrained("./my-adapter")

保存后磁盘上多了一个文件夹，里面只有两个文件：

my-adapter/ ├── adapter_model.safetensors ← 所有 LoRA 的 A 和 B 矩阵权重（~20MB） └── adapter_config.json ← 你写的 LoraConfig 被序列化成 JSON
这两个文件分别存了什么：

adapter_model.safetensors（~20MB）：
就是一个字典：{"q_proj.lora_A.weight": tensor, "q_proj.lora_B.weight": tensor, ...}
64 个层 × 每个层一个 A 和一个 B = 128 个小矩阵

adapter_config.json（~1KB）：
{"r": 16, "lora_alpha": 32, "target_modules": ["q_proj", "v_proj"], ...}
就是你的 LoraConfig 的序列化。加载时 peft 靠它知道怎么重建 LoRA 结构。
关键理解：
保存的不是完整模型（14GB），而是只有 LoRA 的增量（20MB）。
原始模型没有变也没有被保存。你可以把这个 20MB 的 adapter 发给别人，
只要对方有同样的 base 模型就能用。

加载 — base 模型 + adapter 文件 = 可用的模型

部署或继续使用时，需要重新加载。分两步：

from peft import PeftModel from transformers import AutoModelForCausalLM # 第一步：加载原始的 base 模型 base = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-hf") # 第二步：把 adapter "装上去" model = PeftModel.from_pretrained(base, "./my-adapter")

第二步的 PeftModel.from_pretrained 内部做了三件事：

from_pretrained 内部的三步：

① 读配置 — 打开 adapter_config.json，重建 LoraConfig 对象
（它知道了 r=16, target_modules=["q_proj", "v_proj"] 等信息）

② 注入空 LoRA — 调用 get_peft_model(base, config)
（此时 A 和 B 都是随机初始化的，还没有加载训练好的权重）

③ 加载权重 — 读 adapter_model.safetensors，把训练好的 A/B 值填进去
（现在模型和训练结束时完全一样了）
一句话总结：
PeftModel.from_pretrained = get_peft_model（注入空 LoRA）+ load_state_dict（填入训练好的值）

所以你只需要记住：
• 训练时用 get_peft_model（从零开始注入）
• 加载时用 PeftModel.from_pretrained（注入 + 加载已有权重）

merge_and_unload — 把 LoRA 合并回去 + 总结

如果你不再需要 adapter 的灵活性，可以把 LoRA 永久合并到原始权重中：

# 合并 — LoRA 消失，W 变成 W' = W + (α/r)·B·A merged_model = model.merge_and_unload() # 现在 merged_model 是普通的 LlamaForCausalLM，不再需要 peft merged_model.save_pretrained("./merged-model") # 完整模型 ~14GB
什么时候用 merge：
• 部署时 — 不需要 peft 依赖，用普通 transformers 就能加载，推理速度更快
• 量化时 — 合并后再做 GPTQ/AWQ 量化

什么时候不要 merge：
• 还要继续训练这个 adapter
• 同一个模型挂多个 adapter（聊天/代码/数学各一个），需要切换
• 想保持 20MB 的小文件，方便分发
第二轮总结 — 你现在会写的代码：

config = LoraConfig(r, lora_alpha, target_modules) — 写配置单
model = get_peft_model(model, config) — 注入 LoRA
model.print_trainable_parameters() — 检查注入结果
正常的训练循环（不用改任何代码）
model.save_pretrained(path) — 保存 adapter（~20MB）
PeftModel.from_pretrained(base, path) — 加载 adapter
model.merge_and_unload() — 合并回普通模型
