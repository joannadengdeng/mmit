MoReS vis_mask 实现细节 — 图解

理论上 vis_mask 应该精确标记视觉 token, 但实际代码里它一直是 None。最终为什么还能 work？

🎯 理论目的：vis_mask 应该长什么样

MoReS 论文说"只对视觉 token 做干预"。在长度 620 的 hidden states 里，vis_mask 应该这样：

F 系统
T T T T T ... T (576 个 True) 576 个视觉 token
0~29
30,31
32~607 (576 visual tokens)
608~617
618,619

vis_mask 应该是：[F]×30 + [F]×2 + [T]×576 + [F]×10 + [F]×2 — 一个长 620 的布尔张量

然后 MoReS 用 vis_mask 找出视觉位置，从那 576 个里选 norm top 1% (= 6 个) 做干预。

💥 实际代码里发生了什么

# mores.py:332-339 真实代码
def hook(module, input, output):
    h = output[0]                                    # shape: (1, 620, 4096)
    vis_mask = None
    if method_ref._vis_only and image_token_id is not None:
        ids = getattr(model, "_mores_input_ids", None)   # shape: (1, 45) ← input_ids 长度 45
        if ids is not None and ids.shape[1] == h.shape[1]:
            #                                    ↑ 45         ↑ 620
            #                                    永远不相等! 这个判断永远是 False!
            vis_mask = (ids == image_token_id)
    # → vis_mask 一直是 None
关键 bug/feature：因为 input_ids 长度是 45（<image> 还是 1 个 token）， 而进入 hook 时 hidden_states 长度已经是 620（<image> 被展开成 576 个）。 两个 shape 不相等，if 永远不进，vis_mask 一直是 None。

🔄 vis_mask=None 时实际怎么工作

看 OrthogonalIntervention.forward() 里的分支：

def forward(self, h, vis_mask=None, steer_ratio=1.0):
    # selective steering 分支
    for b in range(h.size(0)):
        if vis_mask is not None:
            vis_idx = vis_mask[b].nonzero()[0]
        else:
            vis_idx = torch.arange(h.size(1), device=h.device)
            #                       ↑ 全部 620 个位置都是"候选"

        # top-k by norm
        if steer_ratio < 1.0 and vis_idx.numel() > 1:
            h_vis = h[b, vis_idx]                  # (620, 4096)
            norms = h_vis.norm(dim=-1)            # (620,)
            k = max(1, int(620 * 0.01))      # k = 6
            _, topk_local = norms.topk(k)          # 选 norm 最大的 6 个
            vis_idx = vis_idx[topk_local]
        # 对这 6 个位置做干预

📊 关键问题：从 620 个里选 6 个会选到视觉 token 吗？

观察：norm.topk(6) 选出的 6 个位置（红框）全部落在 visual token 区间。 为什么？因为视觉 token 经过 CLIP+projector 后没有归一化，norm 普遍 5~10， 而文本 token 的 norm 通常稳定在 3~5。所以哪怕从全部 620 个位置里选，topk 也会自然偏向视觉 token。

🎲 三种情况对比

📝 一句话总结

理论：vis_mask 应该长 620, 标记 576 个视觉位置为 True, 然后只从这些位置里挑 norm top 1%。
实际：因为 input_ids (45) 和 hidden states (620) 长度不匹配, vis_mask 永远构造不出来。代码退化成"从全部 620 个位置里选 norm top 1%"。
为什么没出大错：视觉 token 的 norm 通常远大于文本 token, norm.topk(6) 自然就选到视觉位置上, 效果和理论想要的差不多。
论文里要不要说？如果你在写 MoReS 的 follow-up paper, 这是一个值得验证的细节—— 应该实测在 LLaVA-1.5 上, norm.topk(6) 选出的 6 个位置是不是真的全在 32~607 这个 visual 区间。
