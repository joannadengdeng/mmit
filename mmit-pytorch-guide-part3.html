Day 5 — PEFT 参数高效微调方法

5.1 LoRA — 低秩适配 (Low-Rank Adaptation)

核心问题：大模型的线性层权重矩阵 W 通常是 4096×4096 = 16M 个参数。全量微调需要为每个参数计算梯度，存储优化器状态，内存消耗巨大。

核心思想：研究发现，微调时权重的变化量 ΔW 近似低秩。与其学习 16M 个参数的 ΔW，不如直接学习两个小矩阵 A (r×4096) 和 B (4096×r)，其中 r=8。可训练参数量：2 × 8 × 4096 = 65K，仅为原来的 0.4%。

数学公式：

# LoRA 的核心等式
W' = W + (α/r) × B @ A

# W:  原始权重，冻结（不计算梯度）
# A:  低秩矩阵，shape=(r, d_in)，Kaiming uniform 初始化
# B:  低秩矩阵，shape=(d_out, r)，零初始化
# α/r: 缩放因子，例如 α=16, r=8 → scale=2
# B 初始化为零 → 训练开始时 ΔW = 0 → 模型行为不变

最终输出，等价于用 W' = W+(α/r)BA 做线性变换

推理时合并：训练完成后，可以直接计算 W_merged = W + (α/r)×B@A，得到一个普通线性层。推理时零额外延迟。

LoRA 在 Transformer Block 中的位置
Input ↓ ┌───────────────────┐ | Multi-Head Attn | | ┌──────────────┐ | | | q_proj [+LoRA]| | ← LoRA 适配器 | | k_proj [+LoRA]| | ← LoRA 适配器 | | v_proj [+LoRA]| | ← LoRA 适配器 | | o_proj [+LoRA]| | ← LoRA 适配器 | └──────────────┘ | └───────────────────┘ ↓ + residual ┌───────────────────┐ | MLP | | ┌──────────────┐ | | |gate_proj[+LoRA]| ← LoRA 适配器 | |up_proj [+LoRA]| ← LoRA 适配器 | |down_proj[+LoRA]| ← LoRA 适配器 | └──────────────┘ | └───────────────────┘ ↓ + residual Output

LLaVA-1.5-7B 参数计算：7 个目标模块 × 32 层 × 2 矩阵 (A+B) × r(8) × d(4096) = 约 3.6M 可训练参数。相比 7B 全量参数，仅占 0.05%。

5.2 QLoRA — 量化 + LoRA

核心思想：先把基座模型量化到 4-bit（NF4 格式），再在上面叠加 LoRA 适配器（保持 fp16）。这样基座模型从 14GB 压缩到 3.5GB，大幅降低显存需求。

NF4 (Normal Float 4-bit)：专门为正态分布的权重设计的 4-bit 编码。将权重按块分组（通常 64 个一组），找到最优的 16 个量化点（4-bit = 2^4 = 16 个级别），将每个权重映射到最近的量化点。

双重量化 (Double Quantization)：每个量化块需要一个 fp32 的缩放常数。双重量化将这些缩放常数再量化一次，进一步节省内存。

显存对比 (7B 模型)
Full FT: 模型 14GB + 梯度 14GB + 优化器 28GB = 56GB
LoRA: 模型 14GB + LoRA 8MB + 优化器 24MB ≈ 15GB
QLoRA: 模型 3.5GB + LoRA 8MB + 优化器 24MB ≈ 5GB
Full FT 需要 A100 80GB ×1 | LoRA 需要 RTX 4090 24GB ×1 | QLoRA 可在 RTX 3060 12GB 上运行
PyTorch / BitsAndBytes 实现
from transformers import BitsAndBytesConfig
import torch

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                     # 启用 4-bit 量化
    bnb_4bit_quant_type="nf4",              # NF4 量化类型
    bnb_4bit_compute_dtype=torch.float16,   # 计算时反量化到 fp16
    bnb_4bit_use_double_quant=True,         # 双重量化
)
model = AutoModelForCausalLM.from_pretrained(
    model_name, quantization_config=bnb_config
)
# 然后正常应用 LoRA
model = get_peft_model(model, lora_config)

速度代价：QLoRA 的每次前向传播都需要反量化（4-bit → fp16），比纯 fp16 的 LoRA 慢约 30-40%。这是用时间换空间的权衡。

mmit 中的 QLoRA：mmit 支持在 BitsAndBytes 量化模型上叠加 LoRA。对于消费级 GPU（如 RTX 3090），这是训练 7B VLM 的实用方案。

5.3 DoRA — 权重分解 LoRA (Weight-Decomposed LRA)

核心观察：全量微调会同时改变权重的幅度（magnitude）和方向（direction），而标准 LoRA 把两者混在一起。DoRA 将它们分开处理。

# 将权重分解为幅度 m 和方向 V/||V||

# V:  方向矩阵，用 LoRA 修改: V' = V + ΔV = V + (α/r)×BA

效果：相比标准 LoRA 平均提升约 0.7%，推理时可以合并（和 LoRA 一样零额外延迟）。

PyTorch / PEFT 实现（仅需一行改动）

5.4 Mixture of Experts (MoE) — 专家混合

核心思想：不用单个 MLP 处理所有 token，而是准备 N 个"专家"MLP + 一个路由器。每个 token 由路由器选择 top-k 个最合适的专家处理。参数量增大但每个 token 的计算量不变（只激活 k 个专家）。

路由器：一个简单的线性层 + softmax，将输入映射到 N 个专家的概率分布，选择概率最高的 k 个。

MoE 路由流程（N=4 个专家，top-k=2）

输入向量
→

softmax(x @ W_r)
→
得分: [0.45, 0.05, 0.12, 0.38]
选择 top-2:

得分 0.45 (选中)
Expert 1
得分 0.05 (跳过)
Expert 2
得分 0.12 (跳过)
Expert 3
得分 0.38 (选中)
输出 = 0.54 × Expert0(x) + 0.46 × Expert3(x) （权重重新归一化：0.45/(0.45+0.38), 0.38/(0.45+0.38)）

负载均衡损失：如果不加约束，路由器会陷入"赢者通吃"——所有 token 都路由到同一个专家，其他专家完全浪费。需要加辅助损失 L_balance 鼓励均匀分配：

# 辅助负载均衡损失
L_balance = N × Σ_i (f_i × P_i)
# f_i = 分配到专家 i 的 token 比例
# P_i = 路由到专家 i 的平均概率
# 当均匀分配时，f_i = P_i = 1/N，L_balance 最小
MoLE 路由器实现（简化版）
class MoLERouter(nn.Module):
    def __init__(self, hidden_dim, num_experts):
        super().__init__()
        self.gate = nn.Linear(hidden_dim, num_experts, bias=False)

    def forward(self, x):
        scores = self.gate(x)                    # (batch, seq, num_experts)
        probs = torch.softmax(scores, dim=-1)    # 概率分布
        top_val, top_idx = probs.topk(1, dim=-1) # 选择最佳专家
        return top_idx, top_val                   # 专家索引 + 权重
mmit 中的 MoLE：MoLE 在 MLP 层使用多个 LoRA "专家"，在 Attention 层使用标准 LoRA。路由器为每个 token 选择一个 LoRA 专家。这样在不增加推理成本的前提下增强了模型容量。

5.5 Forward Hooks — 不改代码注入行为

核心思想：PyTorch 允许你注册一个函数，在某个模块的 forward() 执行之前或之后自动运行。你可以读取或修改输入/输出，而无需修改模型源代码。

这正是 MoReS 和 ReFT 的工作原理——它们不碰模型代码，而是"挂钩"到已有层上。

Hook 在前向传播中的执行时机
输入 x ↓ [pre_hook(module, input)] ← register_forward_pre_hook: 可修改输入 ↓ [module.forward(x)] ← 模块的正常前向传播 ↓ [post_hook(module, input, output)] ← register_forward_hook: 可修改输出 ↓ 输出 y (可能已被 hook 修改)

两种 Hook 类型：

类型注册方法签名用途
后置钩子 (post-hook)register_forward_hook(fn)fn(module, input, output) → modified_output修改层的输出（MoReS 用此干预表示）
前置钩子 (pre-hook)register_forward_pre_hook(fn)fn(module, input) → modified_input修改层的输入（捕获 input_ids 等）

闭包工厂模式（重要的 Python 陷阱）：

# 错误写法：循环变量捕获 bug
for i in range(9):
    layer.register_forward_hook(
        lambda mod, inp, out: intervene(out, i)   # BUG! i 永远是 8
    )

# 正确写法：用闭包工厂
def make_hook(layer_idx):
    def hook(mod, inp, out):
        return intervene(out, layer_idx)   # layer_idx 在创建时绑定
    return hook

for i in range(9):
    layer.register_forward_hook(make_hook(i))  # 每个 hook 捕获不同的 i
PyTorch Hook API
# 注册钩子，返回 handle
handle = layer.register_forward_hook(hook_fn)

# 移除钩子
handle.remove()

# 实际例子：打印每层输出的统计信息
def debug_hook(module, input, output):
    print(f"output mean={output.mean():.4f}, std={output.std():.4f}")
    return output  # 返回 None 或原始 output 表示不修改

handle = model.layers[0].register_forward_hook(debug_hook)
mmit 中的 Hook：MoReS 在 LLM 的 9 个层上注册 post-hook，用于干预隐藏表示。ReFT 也使用 hook 实现表示微调。Pre-hook 用于捕获 input_ids 以确定干预位置。

Day 6 — 进阶技术

6.1 Mixed Precision Training — 混合精度训练

核心思想：前向和反向传播使用低精度（fp16/bf16），速度快、省内存；优化器状态保持 fp32，保证数值稳定。"混合"精度就是指在同一次训练中同时使用两种精度。

数据类型字节数指数位尾数位数值范围特点
float324823±3.4×10^38标准精度，优化器状态用此
float162510±6.5×10^4范围小，训练易溢出
bfloat16287±3.4×10^38与 fp32 同范围，精度略低，训练首选
bf16 vs fp16 对比
bf16: [1][8位指数][7位尾数]
← 范围大 (同 fp32)，精度低
fp16: [1][5位指数][10位尾数]
← 范围小 (易溢出)，精度高
fp32: [1][8位指数][23位尾数]
← 完整精度，内存 ×2

为什么 bf16 优于 fp16：fp16 的指数只有 5 位，最大值约 65504。训练中梯度或激活值一旦超过这个范围就会变成 inf（溢出）。bf16 与 fp32 有相同的指数范围，几乎不会溢出，所以训练更稳定。

PyTorch 混合精度用法
# 方法一：加载模型时指定 dtype（mmit 的做法）
model = AutoModelForCausalLM.from_pretrained(
    model_name, torch_dtype=torch.bfloat16     # 模型权重用 bf16
)

# 方法二：使用 autocast（自动选择最优精度）
with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
    outputs = model(input_ids)     # 自动使用 bf16 计算
    loss = outputs.loss            # loss 也是 bf16
loss.backward()                     # 梯度也是 bf16
optimizer.step()                    # 优化器内部用 fp32 更新
mmit 中的混合精度：mmit 所有模型均使用 torch_dtype=torch.bfloat16 加载。这意味着模型权重、前向/反向计算全部使用 bf16，优化器状态保持 fp32。这样 7B 模型的内存占用减半，同时训练稳定。

6.2 Label Smoothing — 标签平滑

核心问题：标准的交叉熵损失要求模型输出"硬"标签——对正确类别的概率尽可能接近 1.0，其他类别接近 0.0。这导致模型过度自信（overconfident），泛化能力下降。

解决方案：将硬目标平滑为软目标，给每个非正确类别分配一点概率 ε/C。

标签平滑效果 (ε=0.1, C=4 个类别)
硬标签:
0.0
0.0
1.0
0.0
软标签:
0.025
0.025
0.925
0.025
公式: y_smooth = (1-ε) × y_hard + ε/C    其中 ε=0.1, C=4 
# PyTorch 内置标签平滑
loss_fn = nn.CrossEntropyLoss(label_smoothing=0.1)

# 等价的手动实现
def label_smoothing_loss(logits, targets, epsilon=0.1):
    C = logits.size(-1)                        # 类别数
    log_probs = F.log_softmax(logits, dim=-1)
    nll = -log_probs.gather(dim=-1, index=targets.unsqueeze(-1))
    smooth = -log_probs.mean(dim=-1)
    loss = (1 - epsilon) * nll.squeeze(-1) + epsilon * smooth
    return loss.mean()
mmit 中的标签平滑：mmit 的 losses 目录中包含标签平滑的损失变体，可用于训练时降低模型对 token 预测的过度自信。

6.3 Contrastive Loss (InfoNCE) — 对比学习损失

核心思想：给定一批 (image, text) 配对，让匹配对的相似度尽可能高，不匹配对的相似度尽可能低。这就是 CLIP 的训练方式，也是 LLaVA 视觉编码器的来源。

# InfoNCE 损失公式
L = -log( exp(sim(q, k+) / τ) / Σ_j exp(sim(q, k_j) / τ) )

# q:    查询（如图像嵌入）
# k+:   正样本（匹配的文本嵌入）
# k_j:  所有样本（batch 中所有文本嵌入）
# τ:    温度参数（通常 0.07），越小 → 分布越尖锐
# sim:  余弦相似度
CLIP 对比学习：batch=4 的相似度矩阵
text_0 text_1 text_2 text_3 image_0 [ 0.92 0.11 0.05 0.13 ] ← 对角线是匹配对 image_1 [ 0.08 0.87 0.15 0.03 ] 应尽可能大 image_2 [ 0.12 0.04 0.91 0.07 ] image_3 [ 0.06 0.14 0.09 0.88 ] ← 非对角线应尽可能小 Loss_i2t = -log(exp(diag/τ) / row_sum(exp/τ)) 图像→文本方向 Loss_t2i = -log(exp(diag/τ) / col_sum(exp/τ)) 文本→图像方向 L_clip = (Loss_i2t + Loss_t2i) / 2
PyTorch CLIP 对比损失实现
def clip_loss(image_embeds, text_embeds, temperature=0.07):
    # 归一化
    image_embeds = F.normalize(image_embeds, dim=-1)
    text_embeds = F.normalize(text_embeds, dim=-1)

    # 相似度矩阵 (batch, batch)
    logits = image_embeds @ text_embeds.T / temperature

    # 标签: 对角线 = 匹配对
    labels = torch.arange(logits.size(0), device=logits.device)

    # 双向交叉熵
    loss_i2t = F.cross_entropy(logits, labels)       # 行方向
    loss_t2i = F.cross_entropy(logits.T, labels)     # 列方向
    return (loss_i2t + loss_t2i) / 2
与 mmit 的关系：LLaVA 使用的视觉编码器 (CLIP ViT) 就是用对比损失预训练的。理解 InfoNCE 有助于理解视觉特征的性质——它们被训练为在嵌入空间中与对应的文本描述"靠近"。

6.4 Activation Checkpointing — 用计算换内存

核心问题：反向传播需要用到前向传播时保存的中间激活值。一个 7B 模型的 32 层 Transformer，中间激活可能占用 10+ GB 内存。

解决方案：前向传播时不保存中间激活值，反向传播到某一层时重新计算该层的前向结果。

普通训练 vs 激活检查点
普通训练: Forward: Layer0 → [save act0] → Layer1 → [save act1] → ... → Layer31 → [save act31] Backward: 用 act31 计算 grad31 → 用 act30 计算 grad30 → ... → 用 act0 计算 grad0 内存: 保存全部 32 层的激活 = 很大 激活检查点: Forward: Layer0 → [丢弃] → Layer1 → [丢弃] → ... → Layer31 → [只保存输出] Backward: 需要 act30? → 重跑 Layer30 前向 → 得到 act30 → 计算 grad30 → 丢弃 act30 内存: 只保存少量检查点 = 约 60% 更少 代价: 约 2x 前向计算量
PyTorch 激活检查点 API
from torch.utils.checkpoint import checkpoint

# 对某个函数启用激活检查点
def forward(self, x):
    # 不保存 transformer_block 的中间激活
    # 反向传播时会重新计算
    x = checkpoint(self.transformer_block, x, use_reentrant=False)
    return x

# Hugging Face 模型：一键启用
model.gradient_checkpointing_enable()
# 等价于对每个 Transformer 层启用 checkpoint
方案内存占用计算量适用场景
普通训练高（保存所有激活）1x内存充足时
激活检查点低（约 60% 更少）∼2x 前向内存紧张，GPU 算力有余
检查点 + QLoRA最低∼2.5x消费级 GPU 训练大模型
mmit 中的激活检查点：当在消费级 GPU 上训练时，可以结合 gradient_checkpointing 和 QLoRA 来大幅降低内存需求。model.gradient_checkpointing_enable() 对 HuggingFace 模型一键生效。
