peft · 第一轮：直觉与概念

不写代码，先把 LoRA 和 peft 库在做什么搞懂。看完这轮再去看第二轮的 API 就不跳了。

问题：全量微调太贵了

你有一个预训练好的大模型（比如 LLaMA-7B），想让它学会做新任务。最直接的办法是全量微调：把所有 70 亿个参数都拿来训练。

全量微调的代价：
• 模型参数 7B × 2 字节（float16）= 14 GB
• 梯度也要存：又 14 GB
• Adam 优化器存两个状态（m 和 v）：又 28 GB
• 总共需要 ~56 GB 显存，一张 A100 80GB 才刚刚够

而且训练完的模型也是完整的 14 GB — 你为每个任务存一份完整模型？

LoRA 的想法：别动原始的 70 亿参数了。给模型旁边加一些很小的可训练参数（比如 800 万个），只训练这些小参数，效果也能很好。

LoRA 的核心直觉

模型里有很多 Linear 层（线性变换层），每一层就是一个权重矩阵 W。比如 Attention 的 Q 投影层 q_proj，它的 W 是 4096×4096。

LoRA 的核心思想：

微调时，W 的变化量 ΔW 其实是"低秩"的。

什么意思？就是说 ΔW 这个 4096×4096 的大矩阵，可以用两个小矩阵相乘来近似：
ΔW ≈ B × A
其中 A 是 r×4096，B 是 4096×r，r 很小（比如 16）

原来要存 4096×4096 = 1600万 个参数的变化
现在只需要存 16×4096 + 4096×16 = 13万 个参数
压缩了 120 倍！

打个比方：你不需要重新画一幅完整的画（改 W），只需要在画上面贴一张半透明贴纸（A × B），画面就变了。

r（rank）到底是什么

r 就是 A 矩阵的行数 / B 矩阵的列数，也叫"秩"。它控制 LoRA 的大小。

r 小（比如 4~8）：
• 参数很少，省显存
• 表达能力有限
• 适合简单任务
r 中等（比如 16~32）：
• 最常用的范围
• 效果接近全量微调
• 参数量是 W 的 ~1%
r 大（比如 64~128）：
• 参数更多
• 接近全量微调的效果
• 但省显存的优势减弱
一个具体的算术例子：
LLaMA-7B 有 32 层 attention，每层有 q_proj 和 v_proj 两个层要加 LoRA：

• 每个 LoRA：r=16 → A (16×4096) + B (4096×16) = 131,072 个参数
• 总共 32 层 × 2 个 = 64 个 LoRA
• 64 × 131,072 = ~840 万个可训练参数
• 占原始 70 亿参数的 0.12%

保存到磁盘只需要 ~20 MB（对比原始模型 14 GB）

一个 Linear 层加了 LoRA 后变成什么样？

这是最关键的一步。加 LoRA 之前和之后，一个 Linear 层的计算过程变了：

原来（只有 W）：
输出 = W × 输入

W 是 4096×4096 的大矩阵
全部冻结，不训练
加了 LoRA 之后：
输出 = W × 输入 + B × A × 输入

W 还是冻结的
A 和 B 是新加的小矩阵，只训练它们
理解这个公式，你就理解了 LoRA 的全部：

• W × 输入 — 原始模型的计算，不变
• B × A × 输入 — LoRA 加的"补丁"，这个是可训练的
• 两个结果相加就是最终输出

所以 LoRA 不是替换了原来的层，而是在旁边加了一条小旁路。
原路不动，旁路可调。

模型有那么多层，给哪些加 LoRA？

一个 Transformer 模型有很多层。不是每个层都需要加 LoRA。

Transformer 一层里有这些 Linear 层：

Attention 部分：
• q_proj — 计算 Query
• k_proj — 计算 Key
• v_proj — 计算 Value
• o_proj — Attention 输出投影

FFN/MLP 部分：
• gate_proj、up_proj、down_proj
通常只给 q_proj 和 v_proj 加 LoRA：

• 这是 LoRA 原始论文的推荐做法
• Attention 的 Q 和 V 投影层对任务适应最敏感
• 只改这两个就够了，效果接近全量微调
• 参数量最少，最省显存

当然你也可以选择给更多层加 LoRA（比如也给 k_proj 和 o_proj 加），参数多了效果会好一点，但边际收益递减。

而且 LLaMA-7B 有 32 层这样的结构（32 个 decoder layer），每层都有 q_proj 和 v_proj。所以是 32 层的 q_proj + 32 层的 v_proj = 64 个 Linear 层同时加 LoRA。

peft 库到底在做什么？

你现在知道了 LoRA 的原理。但手动去模型里找到每个 q_proj 和 v_proj，然后给它们加上 A 和 B 矩阵，这太麻烦了。

peft 库就是一个自动注入工具。你告诉它"我要给哪些层加 LoRA，r 设多少"，它帮你：

peft 做的三件事：

① 注入 — 找到你指定的层（比如所有 q_proj 和 v_proj），把它们从普通的 Linear 替换成带 LoRA 旁路的 lora.Linear

② 冻结 — 把原始模型的所有参数设为"不训练"（requires_grad=False），只有新加的 A 和 B 是"要训练的"

③ 包装 — 把修改后的模型包装成一个 PeftModel 对象，它提供保存/加载/合并等便利方法
类比：
模型是一台机器。peft 是一个改装师。

你给改装师一张配置单（"我要给 q_proj 和 v_proj 加 LoRA，r=16"），
改装师打开机器，把指定零件旁边装上小配件（A 和 B），然后锁死原来的零件（冻结），
最后把改好的机器装进一个新外壳里交给你（PeftModel）。

"注入"具体是什么意思？

PyTorch 的模型本质上是一棵模块树。每个模型是很多模块嵌套的：

LlamaForCausalLM └─ LlamaModel └─ layers (32 个) └─ LlamaDecoderLayer ├─ self_attn │ ├─ q_proj: Linear(4096, 4096) ← 这个要改 │ ├─ k_proj: Linear(4096, 4096) │ ├─ v_proj: Linear(4096, 4096) ← 这个要改 │ └─ o_proj: Linear(4096, 4096) └─ mlp ├─ gate_proj: Linear(4096, 11008) ├─ up_proj: Linear(4096, 11008) └─ down_proj: Linear(11008, 4096)

peft 的"注入"就是遍历这棵树，找到名字里包含 q_proj 和 v_proj 的节点，把它们从普通 Linear 替换成 lora.Linear：

替换前：q_proj = Linear(4096, 4096) — 只有一个 W
替换后：q_proj = lora.Linear — 内部有 W（冻结）+ A（训练）+ B（训练）

从外面看，它的输入输出形状没变（还是接收 4096 维输入，输出 4096 维），
但内部多了一条 LoRA 旁路。

训练时发生了什么？

LoRA 注入后的训练和普通训练写法完全一样（forward → loss → backward → step），但内部有很大不同：

前向传播（forward）：
每个 lora.Linear 层同时算两条路：
输出 = W × 输入 + (α/r) × B × A × 输入

α/r 是一个缩放系数（后面的轮次会详细讲），现在只需要知道它控制 LoRA 的影响力大小。
反向传播（backward）：
• 梯度会流过 A 和 B（因为 requires_grad=True）
• 梯度不会流过 W（因为 requires_grad=False）
• 所以 W 的梯度不用计算也不用存储 → 省显存
• optimizer 也只需要给 A 和 B 维护状态 → 又省显存
显存节省来自哪里：
全量微调：参数 14GB + 梯度 14GB + optimizer 状态 28GB = 56 GB
LoRA (r=16)：冻结参数 14GB（只推理不存梯度）+ LoRA 参数/梯度/状态 ~0.06 GB ≈ 14.1 GB

省了 75% 的显存！一张 24GB 的 4090 就能跑。

Adapter：可拆卸的小零件

训练完后，你需要保存结果。LoRA 的一大好处是：只需要保存 A 和 B 矩阵，不需要保存完整模型。

保存的内容（叫 "adapter"）：

所有 64 个 LoRA 的 A 和 B 矩阵权重 → 一个 ~20 MB 的文件
加一个 JSON 配置文件（记录 r=16, alpha=32, target_modules 等）

对比全量微调保存整个模型：14 GB
"Adapter" 这个词的含义：

就像手机壳——手机（原始模型）不变，换个壳（adapter）就变了。

• 你可以给同一个 base 模型做多个 adapter（一个用于聊天，一个用于代码）
• 每个 adapter 只有 20 MB
• 加载时：先加载 base 模型，再把 adapter "装上去"
• 不用了就"拆掉"，base 模型完好无损
还可以合并（merge）：
如果你确定了要用某个 adapter，可以把 LoRA 永久合并回 W：
W_new = W + B × A
合并后变回普通模型，推理速度不受影响，也不再需要 peft 库。

完整流程鸟瞰

把前面讲的串起来，LoRA 的完整流程就是：

① 加载原始模型 — 从 HuggingFace 下载预训练模型（14 GB）

② 配置 LoRA — 告诉 peft："r=16，给 q_proj 和 v_proj 加 LoRA"

③ 注入 — peft 自动把 64 个 Linear 替换成 lora.Linear，冻结原始参数

④ 训练 — 正常写训练循环，只有 LoRA 的 A 和 B 在被更新

⑤ 保存 adapter — 只保存 A 和 B（20 MB），不保存完整模型

⑥ 部署/使用 — 加载 base 模型 + adapter，或者合并后部署
你现在知道了：
✓ 为什么需要 LoRA（全量微调太贵）
✓ LoRA 在做什么（给 Linear 层加 A×B 旁路）
✓ r 是什么（控制旁路大小的超参数）
✓ 哪些层被改了（通常是 q_proj 和 v_proj）
✓ peft 库做了什么（自动注入 + 冻结 + 包装）
✓ "注入"是什么意思（替换模块树的节点）
✓ 训练时梯度怎么流（只流过 A 和 B）
✓ adapter 是什么（可拆卸的小权重文件）
